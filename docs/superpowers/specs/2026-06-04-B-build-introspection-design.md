## 1. Цель + ценность для developer-user

Сейчас build-конвейер Recast наблюдаем **только косвенно**: developer-user может
выбрать один из 16 draw-режимов (`sample_solo.zig:22-39`, `DrawMode`), которые
рисуют промежуточные структуры (`hf`/`chf`/`cset`/`pmesh`/`dmesh`,
`sample_solo.zig:53-57`) в `render()` (`sample_solo.zig:410-447`). Но:

- **Нет понимания «что сделала каждая стадия»** — `doBuild`
  (`sample_solo.zig:225-359`) исполняет 9 стадий (heightfield → filters →
  compact → erode+volumes → regions → contours → polymesh → detail → navmesh)
  единым «чёрным ящиком»; в Log падает только итоговое `Build OK in N ms`
  (`sample_solo.zig:200`). Нет per-stage времени, числа спанов/регионов/
  контуров/полигонов.
- **Нет «до/после»** — поменяв параметр (например `region_min_size`), нельзя
  увидеть дельту: на сколько изменилось число регионов/полигонов, где они
  пропали.
- **Нет диагностики артефактов** — дыры в навмеше, тонкие перешейки,
  деградированные detail-треугольники, регионы-пылинки (отсеянные
  `min_region_area`, `sample_solo.zig:159,272`) developer-user ищет глазами,
  переключая draw-режимы.
- **Инспекция полигона — это `std.debug.print` в stderr** (`main.zig:480-491`):
  выводит лишь `ref/высоту/флаги`, не структурировано, не в UI, без соседей/
  area/площади/links.

**Ценность кластера B:** developer-user, настраивающий параметры сборки
(`CommonSettings`, `FEATURES.md` §3), получает **встроенный «рентген» конвейера**:
шаговый просмотр стадий со счётчиками, наглядную дельту при изменении настроек,
автоматическую подсветку проблемных мест и табличный инспектор полигона. Ответ на
вопрос «**что сломалось/изменилось в билде и почему**» становится встроенным
сценарием, а не ad-hoc разглядыванием draw-режимов и чтением stderr.

Это прямой потребитель **R** (схемы/изоляция/выделение/overlay/легенды, 3.c) и
**U** (панель + реестр, 3.d); читает настройки/геометрию из **Sc** (`settings`,
3.a). **P** — опциональна (○): золотые «снимки счётчиков» для регрессий —
nice-to-have, не ядро кластера.

## 2. Scope (in / out / YAGNI)

### Входит (in)

- **Слайдер/степпер по стадиям** конвейера (heightfield → compact →
  distancefield → regions → contours → polymesh → detail) поверх существующих
  `DrawMode` — единая ось «стадии», синхронизированная с draw-режимом.
- **Per-stage счётчики и тайминги**: на каждую стадию — ключевые метрики (спаны,
  walkable-спаны, регионы, контуры, верты/полигоны, detail-треугольники) и время
  стадии. Сбор — инструментовка-обёртка вокруг `doBuild`, **без правки ядра**.
- **Diff параметров до/после**: хранение метрик предыдущей сборки и наложение
  дельты (Δполигонов/Δрегионов/Δконтуров и т.п.) в UI; опционально — визуальная
  дельта «что пропало/появилось» наложением через **R**.
- **Подсветка артефактов** (детекторы поверх готовых структур): дыры,
  тонкие перешейки, деградированные detail-треугольники, регионы-пылинки
  (`< min_region_area`). Визуализация — через **R.highlight/isolation/color_scheme**.
- **Табличный инспектор полигона**: замена `std.debug.print` (`main.zig:480-491`)
  на UI-панель — area / flags / соседи / площадь / диапазон высот / links
  таблицей. Использует `dt.NavMesh.getTileAndPolyByRef`
  (`src/detour/navmesh.zig:672`), `getPolyFlags` (724), `getPolyArea` (736).

### Не входит (out)

- Изменение faithful-ядра `src/recast/*` ради инструментовки (счётчики собираем
  на demo-уровне обёрткой/пост-обходом структур; см. §4).
- Новые механизмы рендера (схемы/легенды/изоляция/выделение/overlay) — они
  **в фундаменте 3.c** (решение Q7 фундамента). Кластер B их *использует*, а не
  определяет (см. R6 фундамента и §4 тут).
- Партиционирование/фильтры как развилки конвейера — это поведение сэмпла
  (`sample_solo.zig:268-278`); B их **наблюдает**, но не добавляет новых
  (см. `FEATURES.md` §11 — фильтры уже частично заводятся).
- Тайловые/temp-сэмплы как первичная цель: B проектируется на **Solo**
  (`sample_solo.zig`), где есть ВСЕ промежуточные структуры; Tile/Temp — out
  (у Tile только 5 draw-режимов, нет полного набора `hf/chf/cset/...`).

### YAGNI

- Полноценный «diff навмешей» (топологическое сравнение двух собранных
  навмешей) — это кластер G (валидация/регрессии). B делает только **diff
  счётчиков/метрик** + опциональную визуальную дельту регионов/полигонов.
- Профилирование на уровне Tracy/flamegraph — это кластер C. B меряет лишь
  per-stage wall-clock через уже имеющийся `io_util.PerfTimer`
  (`sample_solo.zig:150`).
- Сохранение истории всех сборок / time-travel — храним **только предыдущую**
  сборку для diff (одна дельта). История > 1 — YAGNI.
- Редактирование параметров «на стадии» (пересборка только хвоста конвейера с
  reused-префиксом) — конвейер ядра не разбит на инкрементальные шаги; полная
  пересборка через `build()` — достаточно. YAGNI.
- Артефакт-детекторы как «исправлялки» (авто-фикс дыр) — только диагностика
  (подсветка + счётчик), не мутация. Мутация сцены — кластер F.

## 3. Фичи (приоритезированы)

Приоритет: **P0** (ядро ценности, минимально-полезный продукт кластера) →
**P1** (наглядность) → **P2** (расширения).

### B-1 (P0). Per-stage счётчики и тайминги

- **Что.** Структура `StageStats` с метриками на каждую из 7 стадий + время
  стадии; собирается при сборке. Метрики (из реальных полей ядра):
  - **heightfield**: `getHeightFieldSpanCount` (`sample_solo.zig:256`) — всего
    спанов; walkable-спаны (обход `hf` после фильтров).
  - **compact**: `chf.span_count` (`heightfield.zig:204`), `chf.walkable_height/
    walkable_climb`.
  - **distancefield**: `chf.max_distance` (`heightfield.zig:208`) — только для
    `.watershed` (см. `sample_solo.zig:269-272`; для monotone/layers — N/A).
  - **regions**: `chf.max_regions` (`heightfield.zig:209`); число «живых»
    регионов и сколько отсеяно по `min_region_area`/слито по
    `merge_region_area`.
  - **contours**: `cset.nconts` (`polymesh.zig:39`); суммарно raw-вертов
    (`Σ cont.nrverts`) vs simplified (`Σ cont.nverts`, `polymesh.zig:8-10`).
  - **polymesh**: `pmesh.nverts`/`npolys`/`nvp` (`polymesh.zig:84-87`).
  - **detail**: `dmesh.nmeshes`/`nverts`/`ntris` (`polymesh.zig:142-144`).
- **Зачем.** Базовая наблюдаемость: developer-user видит «сколько чего родила
  каждая стадия» и где время. Фундамент для B-2 (diff) и B-3 (артефакты).
- **UX.** Таблица в новой панели **Build Inspector** (через **U.panel**, 3.d):
  строка-стадия → `[count metrics] [stage ms]`. Итоговое время — сумма (сверить с
  `build_time_ms`, `sample_solo.zig:198`). Лог-дубль в окно Log
  (`bctx.context().log`, `sample_solo.zig:147,200`) опционально.
- **Данные.** Читает: промежуточные `hf/chf/cset/pmesh/dmesh` (Solo держит их,
  `sample_solo.zig:53-57`). Тайминги: per-stage `io_util.PerfTimer.start()/
  readMs()` (паттерн `sample_solo.zig:150,198`).
- **Зависимости.** **Sc**: читает `settings` (для `min_region_area`/
  `merge_region_area` — пороги отсева, `sample_solo.zig:159-160`). **U**: панель.
  Ядро: только **чтение** структур `src/recast/*`. *Сбор счётчиков требует точки
  перехвата по стадиям — см. §4 «точки интеграции» (вариант без правки ядра).*

### B-2 (P0). Diff параметров до/после

- **Что.** Хранить `StageStats` предыдущей успешной сборки; при новой —
  показывать дельту по каждой метрике (Δспанов / Δрегионов / Δконтуров /
  Δполигонов / Δdetail-tris / Δвремя). Плюс снимок параметров (`CommonSettings`),
  чтобы видеть **что именно поменяли**.
- **Зачем.** Прямой ответ «как мой твик параметра отразился на навмеше». Главный
  рабочий цикл настройщика: подвигал слайдер → Build → увидел дельту.
- **UX.** Колонка «Δ» в таблице B-1 (зелёный/красный знак). Под таблицей —
  список изменённых параметров (`region_min_size: 8 → 12` и т.п.), diff по полям
  `CommonSettings`. Кнопка «reset baseline» (забыть предыдущую).
- **Данные.** Два снимка `StageStats` + два снимка `CommonSettings` (значимые
  поля). Diff `CommonSettings` — пофайловое сравнение полей (структура известна,
  `FEATURES.md` §3).
- **Зависимости.** B-1 (источник метрик). **Sc**: snapshot `settings` до/после.
  **R** (опционально, P1-апгрейд): визуальная дельта — «регионы, исчезнувшие
  после ужесточения `min_region_area`» наложением (`R.highlight` + `transCol`).

### B-3 (P1). Подсветка артефактов конвейера

- **Что.** Набор **детекторов** поверх готовых структур (пост-обход, без правки
  ядра), каждый даёт множество «подозрительных» элементов + счётчик:
  - **регионы-пылинки**: при watershed малые регионы отсеиваются внутри
    `buildRegions` по `min_region_area` (`sample_solo.zig:272`), их в финале
    нет — поэтому детектор работает на уровне **compact regions** (по `chf`
    до/в обход span→reg, считаем площади регионов и подсвечиваем те, что
    `< min_region_area`) ИЛИ на контурах (мелкие `cont` по площади). Точную
    точку (chf-regions vs contours) фиксируем в открытом вопросе B-Q3.
  - **деградированные detail-треугольники**: обход `dmesh.tris`
    (`polymesh.zig:140`, layout `[4*ntris]`, под-меши через `dmesh.meshes`
    `[4*nmeshes]`, `polymesh.zig:139`) — вырожденные/иглоподобные треугольники
    (площадь ≈ 0 или min-угол ниже порога).
  - **тонкие перешейки**: на `pmesh` — полигоны/рёбра, где «горлышко» уже
    `2*walkable_radius` (агент физически не пройдёт); эвристика по геометрии
    полигонов `pmesh.verts/polys` (`polymesh.zig:79-80`, layout
    `[maxpolys*2*nvp]`: первая половина — индексы вертов, вторая — соседи).
  - **дыры**: полигоны навмеша/контуры с граничными рёбрами там, где ожидался
    проход (эвристика — изолированные мелкие компоненты связности либо
    граничные рёбра внутри bbox геометрии). Самый эвристичный детектор —
    помечаем как «best-effort», порог настраиваемый.
- **Зачем.** Автоматизирует «глазной» поиск проблем сборки; направляет
  настройщика к местам, которые надо лечить параметрами.
- **UX.** В панели Build Inspector — секция «Artifacts»: чекбоксы-детекторы +
  счётчик найденного на каждый; включение детектора подсвечивает элементы в 3D
  через **R.highlight** (набор полигонов/треугольников) и/или **R.isolation**
  («show only artifacts»). Клик по строке артефакта — фокус камеры (опц., P2).
- **Данные.** Только чтение `chf/cset/pmesh/dmesh`; пороги (`walkable_radius`,
  `min_region_area`, min-угол detail) — из `settings`/констант.
- **Зависимости.** **R** (●●): `highlight.zig`/`isolation.zig`/`color_scheme.zig`
  (3.c) — рисуют наборы. **Sc**: пороги из `settings`. Ядро: чтение структур.
  *Геометрия детекторов — теоретическая работа; при сомнениях по корректности
  эвристик — отдельные под-агенты на каждый детектор (см. §6 R-B3).*

### B-4 (P1). Слайдер/степпер по стадиям

- **Что.** Единая ось «стадия конвейера» (heightfield → compact → distancefield
  → regions → contours → polymesh → detail), управляющая активным draw-режимом и
  выделяющая текущую строку в таблице счётчиков. Маппинг стадия→`DrawMode`:
  heightfield→`.voxels`/`.voxels_walkable`; compact→`.compact`;
  distancefield→`.compact_distance`; regions→`.compact_regions`/
  `.region_connections`; contours→`.raw_contours`/`.both_contours`/`.contours`;
  polymesh→`.polymesh`; detail→`.polymesh_detail` (`sample_solo.zig:22-39`).
- **Зачем.** Превращает 16 разрозненных радиокнопок (`drawDebugMode`,
  `sample_solo.zig:585-610`) в осмысленную «прокрутку конвейера»; синхронизирует
  визуализацию со счётчиками/diff/артефактами.
- **UX.** Слайдер «Stage 1..7» (или ◀/▶) в панели Build Inspector; при движении
  выставляет `solo.draw_mode` (поле публично, `sample_solo.zig:48`; уже меняется
  из `--draw=`, `FEATURES.md` §10) и скроллит таблицу к стадии. Под-варианты
  стадии (например raw/both/contours) — вторичный селектор. **Не заменяет**
  существующие радиокнопки, а дополняет (faithful-режимы остаются).
- **Данные.** Текущая стадия (enum), маппинг на `DrawMode`.
- **Зависимости.** **U** (панель/реестр). Пишет `solo.draw_mode`. **R**
  (опц.): легенда активной стадии (`R.legend`, 3.c).

### B-5 (P0→замена). Табличный инспектор полигона

- **Что.** Замена stderr-инспекции (`main.zig:478-492`, ветка `.none`) на
  UI-таблицу: для полигона под кликом — `ref`, area (`getPolyArea`,
  `navmesh.zig:736`), flags (`getPolyFlags`, `navmesh.zig:724`), соседи
  (из poly-данных тайла через `getTileAndPolyByRef`, `navmesh.zig:672`),
  площадь полигона, диапазон высот (`getPolyHeight`, `navmesh.zig:1663`/
  `query.zig:2269`), links (соседние ref + тип: внутри-тайл/портал/off-mesh).
- **Зачем.** Структурированная инспекция вместо stderr; читаемо, в UI, с полным
  набором полей — для отладки «почему этот полигон такой».
- **UX.** При активном инструменте `.none` (инспекция) клик по навмешу
  заполняет панель «Poly Inspector» (через **U.panel**); выделение полигона в
  3D — через **R.highlight** (`debugDrawNavMeshPoly` уже есть как примитив,
  фундамент 3.c §«что уже есть»). Таблица links: строка на сосед — `neighRef`,
  area, тип ребра.
- **Данные.** `dt.NavMesh` + `ref` (как в `main.zig:480-491`); query для
  `findNearestPoly` уже есть у tester (`main.zig:484`). Доступ к poly-структуре
  тайла — `getTileAndPolyByRef`.
- **Зависимости.** **U** (панель), **R** (выделение). Ядро: только публичные
  геттеры Detour (чтение). **Удаляет** `std.debug.print` из `main.zig:487,489`.

## 4. Архитектура

```
└── build_inspector.zig # UI-панель (U.panel): таблица счётчиков+Δ, секция Artifacts,
                        #   stage-слайдер (B-4), под-панель Poly Inspector (B-5).
                        #   Оркестрирует stage_stats/artifacts/poly_inspector + рисует через R.
```

> `dev/`/`src/`/`test/`, но `demo/src/*` — уже сложившийся GUI-слой (см.
> `sample_solo.zig`, `main.zig`, фундамент создаёт `demo/src/render|shell|
> persist`). Кластер B следует **тому же demo-паттерну фундамента**: новая

### 4.2 Точки интеграции

- **Сбор счётчиков/таймингов (B-1) без правки ядра.** Ключевая развилка
  (см. B-Q2). Два варианта:
  - **(A) Пост-обход (предпочтительно, zero-touch ядра).** После `solo.build()`
    читаем уже сохранённые `solo.hf/chf/cset/pmesh/dmesh` (`sample_solo.zig:

### 4.3 Ссылки на подсистемы фундамента (НЕ передизайн)

| Подсистема | Что берём | Где в фундаменте |
|---|---|---|

## 6. Риски

- **R-B1. Faithful-ядро.** Любой соблазн «добавить счётчик в `buildRegions`/
  `buildContours`» нарушает faithful-порт (CLAUDE.md: `src/*` 1-в-1 с C++).
  *Митигация:* счётчики — ТОЛЬКО пост-обход или demo-хук в `doBuild`
  (`sample_solo.zig`), ядро `src/recast/*` неприкосновенно. Зафиксировано в B-Q2/
  B-Q5.
- **R-B2. Per-stage тайминги недоступны без правки сборки.** Текущий `doBuild` —
  единый блок под одним `PerfTimer` (`sample_solo.zig:150,198`). *Митигация:*
  вариант B (B-Q2) — минимальная вставка `mark()`; если владелец откажет — B-1
  деградирует до counts-only (тайминги опускаем).
- **R-B3. Корректность эвристик артефактов (B-3).** «Тонкий перешеек»/«дыра» —
  не строго определённые понятия; ложные срабатывания подорвут доверие.
  *Митигация:* пороги настраиваемые; «дыра» помечена best-effort; **при
  сомнении в математике каждого детектора — отдельный под-агент на детектор**
  (CLAUDE.md: при неуверенности порождать агентов на конкретную область),
  верификация на тест-мешах (`test_data`).
- **R-B4. Перф пост-обхода/детекторов на больших навмешах.** Обход chf-spans/
  pmesh-polys per-build. *Митигация (= R6 фундамента):* детекторы по запросу
  (чекбокс), не каждый кадр; считать один раз после Build, кэшировать;
  переиспользовать дешёвые предикаты.
- **R-B5. Зависимость от ещё-не-готового фундамента.** B-3/B-5 рендер требуют
  **R** (3.c), панели — **U** (3.d), пороги — **Sc** (3.a). Если фундамент не
  доведён до шага 3 — B блокируется. *Митигация:* B-1/B-2 (counts/diff, чистые
  данные + простая таблица) можно делать **раньше R**, на временной панели в
  `main.zig`; рендер-фичи (B-3/B-4/B-5-highlight) — после R. Явная этапизация §7.
- **R-B6. Раздувание `main.zig` (= R5 фундамента).** Если делать панель инлайн
  в `main.zig` до прихода **U**. *Митигация:* как только `U.panel`/
  `tool_registry` готовы — переносим; до того держим B-код в `introspect/*`,
  в `main.zig` только тонкий вызов.
- **R-B7. Маппинг стадия→DrawMode неоднозначен.** Несколько режимов на стадию
  (contours = raw/both/contours; regions = regions/connections). *Митигация:*
  основной режим на стадию + вторичный селектор под-вариантов (B-4 UX); не
  ломаем существующие радиокнопки.

## 7. Этапы реализации (порядок)

Порядок учитывает зависимость от фундамента (U→Sc→R, foundation §7) и принцип
«сначала данные, потом рендер».

1. **B-1 counts (P0, до R/U).** `stage_stats.zig`: `StageStats.collectFrom(...)`
   пост-обходом готовых структур (`hf/chf/cset/pmesh/dmesh`). Тайминги — по
   решению B-Q2 (вариант B: вставить `probe.mark()` в `doBuild`). Вывод —
   временно простой таблицей (или в Log), без зависимости от R. *Разблокирует
   ценность сразу.*
2. **B-2 diff (P0).** В `stage_stats.zig`: хранение `prev`, `diff()→StageDelta`;
   snapshot `CommonSettings` до/после из **Sc**. UI-колонка Δ + список
   изменённых параметров. Зависит от B-1 + Sc.
3. **B-5 инспектор полигона (P0).** `poly_inspector.zig`: `PolyInfo(ref)` через
   Detour-геттеры; **удалить** `std.debug.print` из `main.zig:487,489`, завести
   вызов в ветке `.none`. Таблица — в панели (через **U**, если готова; иначе
   временно в `main.zig`). Выделение полигона — через **R.highlight** (или
   уже-существующий `debugDrawNavMeshPoly`).
4. **B-4 stage-слайдер (P1).** `build_inspector.zig`: слайдер стадий ↔
   `solo.draw_mode`; панель через **U.panel**. Требует **U**.
5. **B-3 артефакты (P1).** `artifacts.zig`: детекторы по одному
   (dust-regions → degenerate-detail-tris → thin-necks → holes), каждый с
   верификацией (под-агенты на спорные, R-B3). Рендер — через **R.highlight/
   isolation/color_scheme/legend**. Требует **R**.
6. **B-2-визуал + P-снимки (P2, опц.).** Визуальная дельта регионов/полигонов
   (R), золотые снимки `StageStats` через **P** для регрессий. YAGNI до
   подтверждённой потребности (кластер G — основной потребитель регрессий).

Критический путь ценности: **1 → 2 → 3** (counts + diff + инспектор) — даёт
рабочий «рентген билда» даже до полного R/U (на временной панели). **4 → 5**
после готовности U/R фундамента. **6** — по потребности.
