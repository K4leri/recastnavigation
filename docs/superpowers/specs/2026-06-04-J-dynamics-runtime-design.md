## 1. Цель и ценность для developer-user

Сейчас динамика в демо — это два изолированных «живых» сэмпла/инструмента, которые
показывают *что* происходит, но не дают анализировать *почему* и не позволяют
воспроизвести/измерить происходящее:

- **TileCache** (`demo/src/sample_temp_obstacles.zig`): можно ставить/снимать
  цилиндры-препятствия (`addObstacle`/`removeObstacleNear`, строки 257-284),
  tilecache обновляется покадрово (`update`, строка 296). Но: перестройка тайлов
  невидима (нет подсветки «какие тайлы регенерируются»), нет тайм-лайна правок, в
  `applyUpdates` (286-294) обновление прокручивается в один кадр «втихую» — а само
  ядро `TileCache.update` (`src/detour_tilecache/tilecache.zig:799`) обрабатывает
  **по одному тайлу за вызов** через `update_queue`/`nupdate` (866-877), и эта
  очередь — богатый источник debug-информации, который сейчас выбрасывается.
- **Crowd** (`demo/src/tool_crowd.zig`): полноценный порт `Tool_Crowd` с perf-
  графиком (`renderPerfGraph`, 328), VO-облаком, follow-трейлами. Но нет
  записи/реплея сценария (каждый прогон ручной и неповторимый), нет агрегатных
  графиков плотности/застреваний по всей толпе, и главное — нет ответа на вопрос
  **«почему вот этот агент стоит на месте»**, хотя ядро уже хранит всё нужное:
  `target_state` (`crowd.zig:279`, значения `target_failed`/`target_requesting`/…),
  `partial` (262), `state=.invalid` (54), `ncorners` (278), `desired_speed`/`vel`
  (268/273). Эти поля сейчас влияют только на цвет цилиндра (`agentCol`,
  `tool_crowd.zig:339-348`) — не на диагностику.

**Ценность J:** developer-user, отлаживающий *динамический* навмеш (стримящиеся
тайлы, открывающиеся двери, толпа в узком месте), получает: (1) видимую во времени
перестройку tilecache с подсветкой затронутых тайлов; (2) воспроизводимый сценарий
толпы (запись→реплей→сравнение), пригодный для регрессий и для «покажи багу
коллеге»; (3) встроенный диагностический ответ «почему агент не движется» вместо
догадок по цвету; (4) динамические off-mesh (дверь открыть/закрыть) и наблюдение
влияния на пути в реальном времени. Это превращает динамику из «демонстрации» в
**инструмент анализа рантайм-поведения**.

## 2. Scope (in / out / YAGNI)

### Входит (in)

- **TileCache time-line:** журнал событий препятствий (add/remove с временем и
  позицией), пошаговая (а не «вся очередь за кадр») визуализация перестройки с
  подсветкой регенерируемых тайлов поверх R-overlay; счётчики
  `getObstacleCount()`/`nupdate`.
- **Crowd record/replay:** запись детерминированного сценария (seed входных
  событий: add-agent, move-target, set-velocity, toggle-poly, obstacle add/remove)
  + фиксированный `dt`-степпинг; реплей с того же стартового состояния сцены;
  сериализация сценария через P (persist).
- **Crowd-аналитика:** агрегатные графики поверх существующего `ValueHistory`/
  `drawGraph` (`value_history.zig`): плотность (макс. занятость прокси-ячейки —
  данные уже есть в `getGrid().getItemCountAt`, `tool_crowd.zig:397`), число
  «застрявших» агентов (по детектору ниже), средняя/макс. скорость.
- **Crowd «why-stuck» диагностика:** панель/overlay, классифицирующая *почему*
  выбранный (или любой) агент не движется, из уже хранимых полей агента.
- **Динамические off-mesh (дверь):** toggle включения off-mesh-связи в рантайме и
  немедленное наблюдение влияния на crowd-пути (через флаг `disabled` на полигонах
  off-mesh-связи — механизм Toggle Polys уже есть, `tool_crowd.zig:267-278`).

### Не входит (out)

- Замена/переписывание faithful-ядра `src/detour_crowd/*`, `src/detour_tilecache/*`.
  Вся логика J — demo-уровень поверх существующих публичных API.
- Новый физический движок/новая модель агентов (steering — как в ядре).
- Сетевой/multi-user реплей, облачное хранение сценариев (это кластер I).
- Полноценный временной «scrubber» с произвольной перемоткой состояния толпы
  назад (см. YAGNI ниже).

### YAGNI (сознательно откладываем)

- **Перемотка толпы назад по времени (bidirectional time-travel).** Требует
  снапшотов всего состояния `Crowd` каждый кадр (тяжело, ядро не сериализуемо без
  работы). Реплей делаем **только вперёд от старта** через детерминированный
  re-sim из журнала событий — этого достаточно для воспроизводимости. Произвольный
  scrubber — отдельный большой кусок, не сейчас.
- **Сжатие журнала tilecache-событий / компакция.** Объём правок мал (порядок
  десятков-сотен событий за сессию) — пишем append-only без компакции.
- **Профилирование per-tile стоимости регенерации с Tracy-зонами.** Это кластер C
  (перф); J только показывает «какой тайл сейчас строится» и грубое время.
- **Кастомные модели плотности/heatmap-интерполяция.** Берём готовую прокси-сетку
  ядра (`ProximityGrid`), рисуем ячейки как есть (R-overlay), без сглаживания.

## 3. Фичи (приоритезированы)

Приоритет: **P0** — ядро ценности кластера, делаем первым; **P1** — высокая
ценность, после P0; **P2** — приятно иметь.

### P0-1. Crowd «why-stuck» диагностика

- **Что:** для выбранного агента (или для всех в режиме detail-all) демо
  классифицирует причину неподвижности и показывает её текстом (overlay над
  агентом + строка в панели), а также подсвечивает релевантную геометрию
  (например — целевой полигон при `target_failed`).
- **Зачем:** сейчас единственный сигнал «что-то не так» — цвет цилиндра
  (`agentCol`, `tool_crowd.zig:339`); причина (нет пути / частичный путь / агент вне
  навмеша / нет цели / заблокирован соседями / упёрся в препятствие) не
  раскрывается. Это самый частый вопрос при отладке толпы.
- **UX:** новая под-секция в `treeNode("Selected Debug Draw")` — чекбокс
  **Show Why-Stuck**. При включении над неподвижными агентами рисуется короткая
  метка (через `ui.screenText`/R.overlay), в панели — строка с детальной причиной
  для выбранного. Классы причины (выводятся из полей агента, без изменения ядра):
  - `NO_TARGET` — `target_state == .target_none` (`crowd.zig:279`).
  - `OFF_NAVMESH` — `state == .invalid` (54/261).
  - `PATH_FAILED` — `target_state == .target_failed` (93/279).
  - `WAITING_PATH` — `target_requesting`/`waiting_for_queue`/`waiting_for_path`
    (агент ждёт планировщик; держится дольше N кадров → подозрительно).
  - `PARTIAL_PATH` — `partial == true` (262): путь не доходит до запрошенной точки.
  - `ARRIVED` — расстояние до `target_pos` < радиус прибытия (норма, не «застрял»).
  - `BLOCKED_LOCAL` — `target_state == .target_valid`, есть `ncorners` (278), но
    `|vel|` (273) ≈ 0 при `desired_speed` (268) > 0 несколько кадров подряд →
    локально заблокирован (соседи/VO/препятствие). Детектор «стоит, хотя хочет
    идти» — скользящее окно по `vel`.
- **Данные:** только чтение полей `CrowdAgent` (`crowd.zig:259-285`) через
  существующий `getAgent` (978). Детектор `BLOCKED_LOCAL` хранит per-agent
  скользящее среднее `|vel|` — новое demo-состояние (массив на `MAX_AGENTS`).
- **Зависимости:** Sc (нет — читает crowd напрямую); R ● (overlay-аннотации
  `overlay.zig`, подсветка целевого полигона `highlight.zig`); U ● (чекбокс в меню
  инструмента). Ядро — без правок.

### P0-2. TileCache: визуализация перестройки тайлов во времени

- **Что:** показать, какие тайлы tilecache затронуты текущими правками и
  регенерируются прямо сейчас; пошаговое (контролируемое) применение очереди
  обновлений вместо «всё за кадр».
- **Зачем:** ядро `TileCache.update` строит **ровно один тайл за вызов**
  (`tilecache.zig:866-877`, `up_to_date` сигналит «очередь пуста», 914-916). Сейчас
  демо в `applyUpdates` крутит цикл до опустошения очереди **молча** (`sample_temp_obstacles.zig:286-294`)
  — вся ценность пошаговости и видимость «что перестраивается» теряется.
- **UX:** в панели Temp Obstacles — секция **Tile Cache Debug**:
  - режим применения: **Instant** (как сейчас) / **Stepped** (один тайл за кадр,
    с подсветкой) / **Manual** (шаг по хоткею, как `1` в crowd).
  - подсветка затронутых тайлов: жёлтый контур у тайлов в `update_queue`, красный
    — у тайла, строящегося в этом кадре (R.isolation/highlight per-tile).
  - счётчики: `pending tiles: nupdate`, `obstacles: getObstacleCount()`
    (`tilecache.zig:324`), последнее время регенерации одного тайла.
- **Данные:** требуется доступ к `nupdate`/`update_queue` tilecache. **Эти поля
  сейчас приватны для demo-обхода?** — нужен публичный геттер
  `pendingUpdateCount()` и итератор затронутых `(tx,ty,layer)` по
  `CompressedTileRef`. Это **минимальное** добавление read-only геттера в ядро
  (см. Открытый вопрос Q1 — единственное предполагаемое касание `src/*`; faithful-
  поведение не меняется, только экспонируется существующее состояние).
- **Зависимости:** R ● (per-tile подсветка контуром — обобщение
  `debugDrawBoxWire`, уже используется для bounds в `sample_temp_obstacles.zig:333`);
  U ● (секция/режим в панели); P ○ (опц.: см. P1-2). Ядро — только read-only
  геттер (Q1).

### P1-1. Crowd record / replay сценария

- **Что:** записать детерминированную последовательность входных событий толпы и
  воспроизвести её с того же стартового состояния сцены, получая идентичный прогон;
  сохранить/загрузить сценарий файлом.
- **Зачем:** сейчас каждый прогон толпы — ручной и неповторимый. Для регрессий
  (кластер G), для демонстрации бага, для бенч-сравнения steering — нужен
  воспроизводимый сценарий. Crowd при фиксированном `dt` и фиксированном порядке
  событий детерминирован (ядро не использует случайность в стиринге).
- **UX:** новая панель/секция **Crowd Scenario**: кнопки **Rec** (старт записи),
  **Stop**, **Play**, **Save…/Load…**. Во время записи каждое действие инструмента
  (`onClick` в `tool_crowd.zig:207`: add/remove agent, move-target, set-velocity,
  toggle-poly) пишется как событие с **номером кадра** симуляции (а не wall-clock).
  Реплей: reset crowd → проигрывание событий по кадрам с фиксированным
  `dt = 1/20` (как `singleStep`, `tool_crowd.zig:201-205`).
- **Данные:** журнал событий `ScenarioEvent { frame:u32, kind, params }`. Ссылка на
  стартовое состояние сцены — через `SceneMeta.uuid`/версию (Sc 3.a) для проверки,
  что реплей идёт на той же геометрии/навмеше. Сериализация — через P (persist):
  бинарный реестр `scenario.bin` с заголовком magic/version/XXH3 (как edits/*,
  фундамент 3.b). Желательно лежит в `<scene>.recastscene/` рядом со сценой.
- **Зависимости:** Sc ● (`SceneMeta` для привязки сценария к сцене); P ● (формат
  `scenario.bin` поверх `write_atomic`+`checksum`, фундамент 3.b — **зависит от
  ещё-не-реализованного фундамента**, см. этапы); U ● (панель Scenario через
  `panel.zig`); R ○. Ядро — без правок (реплей через публичные API crowd).

### P1-2. Crowd-аналитика (плотность / застревания / скорость)

- **Что:** агрегатные графики по всей толпе во времени поверх существующего
  `value_history.zig`: пиковая плотность (макс. агентов в ячейке прокси-сетки),
  число «застрявших» (детектор из P0-1), средняя и макс. скорость.
- **Зачем:** perf-граф (`renderPerfGraph`, `tool_crowd.zig:328`) показывает
  *стоимость update*, но не *качество поведения* толпы. Для отладки заторов в узких
  местах нужна динамика плотности и счётчик застреваний.
- **UX:** чекбокс **Show Crowd Stats** в `treeNode("Debug Draw")`. Рисует 2-3
  стек-графика тем же `vh_mod.drawGraph` (паттерн `renderPerfGraph`,
  `tool_crowd.zig:334-335`), сэмплируемых раз в кадр в `update`
  (`tool_crowd.zig:298`). Опционально — heatmap плотности поверх прокси-сетки
  (расширение существующего Show Prox Grid, `tool_crowd.zig:382-409`: цвет ячейки
  по `getItemCountAt`, уже есть).
- **Данные:** новые `ValueHistory` поля в `CrowdTool` (как `crowd_total_time`,
  `tool_crowd.zig:62`). Источники: `getGrid().getItemCountAt` (макс по ячейкам),
  детектор `BLOCKED_LOCAL` из P0-1 (счётчик), `ag.vel` (скорости).
- **Зависимости:** R ● (графики/легенды — `value_history.zig`/`legend.zig`); U ●
  (чекбокс); Sc ○; P ○. Ядро — без правок.

### P2-1. Динамические off-mesh связи (дверь открыть/закрыть)

- **Что:** в рантайме включать/выключать конкретную off-mesh-связь («дверь») и
  наблюдать, как меняются пути толпы немедленно.
- **Зачем:** классический геймплейный кейс (дверь, мост, лифт). Демонстрирует
  реакцию пути на изменение топологии без полной пересборки.
- **UX:** в инструменте Crowd — режим/действие **Toggle Door**: ЛКМ по off-mesh-
  связи переключает её доступность. Закрытая дверь рисуется красным, открытая —
  зелёным. Существующие агенты перепланируют путь (ядро crowd само инвалидирует
  путь при смене флагов полигона — `checkPathValidity`, `crowd.zig:1166`).
- **Данные:** off-mesh-связь имеет полигон в навмеше; механизм — тот же флаг
  `disabled`, что уже переключается в Toggle Polys (`tool_crowd.zig:267-278`,
  `getPolyFlags`/`setPolyFlags`). Нужно: найти off-mesh polyref под кликом
  (off-mesh связи в `InputGeom`, рисуются `drawOffMeshConnections`). Состояние
  «открыта/закрыта» — на уровне сцены (Sc) для персиста.
- **Зависимости:** Sc ● (флаг состояния двери в модели сцены); R ● (цвет
  открыта/закрыта); U ● (режим инструмента); P ○ (персист состояния — опц.). Ядро —
  без правок (используем `setPolyFlags`, уже публичный).
- **Приоритет P2:** наименьшая новизна (механизм toggle-poly уже есть), наибольшая
  зависимость от того, как Sc/persist оформят off-mesh (3.a отмечает off-mesh как
  «хрупкие 6 параллельных массивов» — лучше делать после стабилизации Sc).

## 4. Архитектура

Новые файлы — все на **demo-уровне** (`demo/src/`), snake_case
(`.agent/project_structure.md`). Инструменты регистрируются через U-shell
(`tool_registry.zig`, фундамент 3.d), визуализация — через R (3.c), персист — через
P (3.b), привязка к сцене — через Sc (3.a). **Ни одну подсистему фундамента J не
передизайнивает** — только потребляет их API.

### 4.1. Новые файлы demo/src

```
demo/src/dynamics/
├── why_stuck.zig        # P0-1. Классификатор причины неподвижности агента из
│                        #   полей CrowdAgent (crowd.zig:259-285). Чистая функция
│                        #   classify(ag) -> StuckReason + per-agent скользящее |vel|.
│                        #   Без состояния ядра; overlay-рендер через R.overlay.
├── tilecache_debug.zig  # P0-2. Режимы применения (Instant/Stepped/Manual) +
│                        #   подсветка update_queue/строящегося тайла. Обёртка над
│                        #   TileCache.update (по 1 тайлу) + чтение pending-геттера (Q1).
├── crowd_stats.zig      # P1-2. Агрегатные ValueHistory (плотность/застревания/
│                        #   скорость) + сэмплинг раз в кадр. Поверх value_history.zig.
├── scenario.zig         # P1-1. Модель сценария: ScenarioEvent{frame,kind,params},
│                        #   рекордер (запись из onClick), плеер (детерм. re-sim
│                        #   с фикс. dt). Чистая модель, без I/O.
└── scenario_io.zig      # P1-1. scenario.bin <-> Scenario через P (write_atomic +
                         #   checksum, magic/version). ЗАВИСИТ от фундамента 3.b.
```

> `door`-логика (P2-1) — НЕ отдельный файл: это режим внутри `tool_crowd.zig`
> (новый `ToolMode.toggle_door`) + поле состояния в Sc. Слишком мал для модуля.

### 4.2. Точки интеграции (существующий код)

- **`tool_crowd.zig`** — основной потребитель:
  - `update` (298): добавить вызовы сэмплинга `crowd_stats` и обновления
    скользящего `|vel|` для `why_stuck`.
  - `render` (350): добавить overlay why-stuck и heatmap плотности (расширение Show
    Prox Grid, 382).

## 5. Открытые вопросы / допущения к владельцу

1. **Q1 — read-only геттер pending-очереди в ядре tilecache (единственное
   касание `src/*`).** P0-2 требует знать, какие тайлы в `update_queue`/`nupdate`
   (`tilecache.zig:866`), чтобы их подсветить. Сейчас это приватное поле.
   *Допущение:* добавить **read-only** `pub fn pendingUpdateCount()` +
   итератор/срез затронутых `CompressedTileRef` — faithful-поведение не меняется,
   только экспонируется существующее состояние. **Подтвердить, что такое
   минимальное добавление в ядро допустимо** (альтернатива — вести «теневую»
   очередь на demo-уровне, дублируя `queryTiles`-логику; хуже, рассинхрон-риск).

2. **Q2 — глубина реплея толпы.** Допущение: реплей **только вперёд от старта**
   (детерм. re-sim из журнала, YAGNI на time-travel назад). **Достаточно ли?** Если
   нужен scrubber назад — это отдельный большой кусок (снапшоты `Crowd`), вынести в
   будущий спек.

3. **Q3 — детерминизм crowd при реплее.** Допущение: ядро crowd детерминировано при
   фиксированных `dt` и порядке событий (нет RNG в стиринге). **Подтвердить, что в
   `src/detour_crowd/*` нет источников недетерминизма** (напр. зависимости от
   порядка итерации хэш-таблиц/указателей). Если есть — реплей будет «похожим», не
   «идентичным»; тогда G-регрессии по толпе невозможны точные.

4. **Q4 — нужен ли персист tilecache-журнала (тайм-лайн правок препятствий между
   сессиями).** Допущение: P0-2 тайм-лайн — **только в рамках сессии** (in-RAM),
   персист правок препятствий — НЕ в J (это поведение P/F-кластеров). **Хотим ли
   сохранять/реплеить последовательность obstacle add/remove durably?** Если да —
   формат аналогичен `scenario.bin`, добавить в этап.

5. **Q5 — модель «двери» (P2-1) в Sc.** Off-mesh в Sc 3.a помечены как «хрупкие 6
   параллельных массивов»; состояние «открыта/закрыта» — новое поле. **Где живёт
   флаг двери: на off-mesh-записи в Sc, или отдельный оверлей-слой?** Допущение: на
   off-mesh-записи (естественно для персиста), но это требует расширения off-mesh-
   модели в Sc — координировать с владельцем 3.a. P2 именно поэтому.

6. **Q6 — приоритет внутри кластера.** Допущение: P0 = why-stuck + tilecache-
   таймлайн (наибольшая аналитическая ценность, минимум зависимостей от ещё-не-
   готового фундамента-P). Record/replay (P1-1) — после фундамента-P. **Согласен?**

## 6. Риски

- **RJ1 — Зависимость P1-1 (record/replay I/O) от ещё-не-реализованного
  фундамента-P (3.b).** `scenario_io` нельзя завершить до `write_atomic`/`checksum`.
  *Митигация:* раздельные `scenario.zig` (модель+реплей в RAM, готов сразу) и
  `scenario_io.zig` (ждёт P); record/replay демонстрируется без save/load на ранней
  стадии.
- **RJ2 — Недетерминизм crowd ломает точный реплей/регрессии (Q3).** Если в ядре
  есть скрытый недетерминизм — реплей не побитовый. *Митигация:* тест детерминизма
  (`test/integration/`): два прогона одного журнала → сравнить траектории; если
  расходятся — задокументировать как «approximate replay», не давать ложных
  гарантий G.
- **RJ3 — Касание ядра (Q1) нарушает «faithful 1-в-1».** Даже read-only геттер —
  отклонение от upstream-структуры. *Митигация:* геттер тонкий, не меняет поведение/
  поля; альтернатива (теневая очередь на demo) хуже по рассинхрону. Решение — за
  владельцем (Q1).
- **RJ4 — Stepped-tilecache меняет ощущаемое поведение демо.** Сейчас обновление
  «мгновенное» (`applyUpdates` крутит до конца). Stepped-режим растягивает его на
  кадры — при большом числе тайлов навмеш «достраивается» заметно. *Митигация:*
  Instant остаётся дефолтом (1-в-1 текущее поведение); Stepped/Manual — opt-in для
  отладки.
- **RJ5 — Раздувание `tool_crowd.zig`/`sample_temp_obstacles.zig`.** Кластер
  навешивает много на два уже немаленьких файла. *Митигация:* логика — в
  `demo/src/dynamics/*`, в существующие файлы только тонкие хуки вызова; панели —
  через U-shell `panel.zig`, не инлайн (фундамент R5).
- **RJ6 — Перф overlay плотности/why-stuck на больших толпах (128 агентов).**
  Per-agent overlay-текст + heatmap каждый кадр. *Митигация:* why-stuck-текст —
  только для неподвижных/выбранного (не для всех 128 поголовно); heatmap
  переиспользует существующий батч Show Prox Grid (`tool_crowd.zig:392`).
- **RJ7 — Привязка сценария к сцене (Sc.uuid).** Реплей на изменённой геометрии
  даёт мусор. *Митигация:* `scenario.bin` хранит `SceneMeta.uuid`+версию; при
  несовпадении — предупреждение в Log, реплей не запускается (или «force»-флаг).

## 7. Этапы реализации (порядок)

Порядок выбран так, чтобы (а) сперва дать максимум ценности на **уже доступном**
ядре/фундаменте-R/U, (б) отложить то, что блокировано фундаментом-P, (в) не ломать
faithful-поведение (Instant-дефолты).

1. **P0-1 — why-stuck (нужен только R+U).** `why_stuck.zig`: чистый
   `classify(ag) -> StuckReason` из полей `CrowdAgent` + per-agent скользящее
   `|vel|`; чекбокс Show Why-Stuck в `tool_crowd.drawMenu`; overlay через R. Тест
   детектора в `test/integration/`. *Разблокирует:* немедленную диагностику толпы,
   ничего не ждёт.
2. **P0-2 — tilecache time-line (R+U, + read-only геттер ядра Q1).** Согласовать
   Q1; `tilecache_debug.zig`: режимы Instant/Stepped/Manual в `applyUpdates`,
   подсветка `update_queue`/строящегося тайла, счётчики. Хоткей Manual-step.
   *Разблокирует:* видимую динамику перестройки.
3. **P1-2 — crowd-аналитика (R+U, на готовом `value_history`).** `crowd_stats.zig`:
   ValueHistory плотности/застреваний/скорости, сэмплинг в `update`, графики в
   `renderPerfGraph`-стиле; опц. heatmap. *Разблокирует:* анализ заторов.
4. **P1-1 (модель) — scenario record/replay в RAM (Sc.SceneMeta + U).**
   `scenario.zig`: рекордер из `onClick`, детерм. плеер с фикс. `dt`; панель
   Scenario (Rec/Stop/Play). Тест детерминизма (Q3). *Разблокирует:*
   воспроизводимость в сессии.
5. **P1-1 (I/O) — scenario save/load (ЖДЁТ фундамент-P 3.b).** `scenario_io.zig`:
   `scenario.bin` (magic `'SCEN'`/version/XXH3) поверх `write_atomic`+`checksum`;
   привязка к `Scene.uuid`. *Разблокирует:* шаринг/регрессии (G/I).
6. **P2-1 — динамические off-mesh «двери» (ЖДЁТ стабилизации Sc off-mesh-модели
   3.a).** `ToolMode.toggle_door` в `tool_crowd`, флаг состояния в Sc, цвет
   открыта/закрыта, реакция путей. *Разблокирует:* рантайм-топологию.

Критический путь ценности: **1 → 2 → 3 → 4**. Шаги 5-6 — по готовности фундамента-P
и стабилизации off-mesh в Sc.
