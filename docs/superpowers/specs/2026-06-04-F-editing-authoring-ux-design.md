## 1. Цель + ценность для developer-user

Сейчас authoring сцены (off-mesh-связи, convex-volumes, area-типы, poly-флаги)
в демо — это **деструктивные, безоткатные** правки прямо в `InputGeom` /
module-global реестрах, с **немедленной полной пересборкой** navmesh:

- `tool_convex.zig:103-121` / `:194-196` — `addConvexVolume`/`deleteConvexVolume`
  без истории; `tool_offmesh.zig:34-44` — `addOffMeshConnection` без истории;
- любая правка взводит `dirty`/`rebuild_needed` (`main.zig:506-532`), и
  пересобирается **вся** сцена (`solo.build()`/`tile.build()`), даже если изменился
  один volume в одном тайле (`sample_tile.zig:60-115` строит **все** tw×th тайлов);
- ввод координат — только мышью по рейкасту меша (`main.zig:470-499`), нет
  числового ввода и привязки к сетке;
- работа поштучная: нет мультивыделения, копирования, группового перемещения
  volumes/connections;
- area-типы и наборы флагов настраиваются по одному в модальном редакторе
  (`tool_convex.zig:219-290`), нет пресетов/шаблонов;
- ошибка правки (не туда кликнул, удалил не тот volume) **необратима** —
  единственный выход «откатить» это вручную восстановить геометрию.

**Цель кластера F** — превратить authoring в *безопасный, точный, групповой и
воспроизводимый* процесс:

- **Безопасный** — полноценный undo/redo для всех правок (volumes / off-mesh /
  area / flags), поверх `Scene` + `Scene.DirtyBits` (фундамент 3.a).
- **Точный** — snap к геометрии/сетке + числовой ввод координат вместо «на глаз».
- **Групповой** — мультивыделение, копирование, перемещение наборов
  volumes/connections.
- **Быстрый** — пересобирать только затронутые тайлы (а не всю сцену) по хоткею,
  через уже существующие `NavMesh.removeTile`/`addTile`/`calcTileLoc`
  (`src/detour/navmesh.zig:1088,1428,373`) и per-tile `buildTileMesh`
  (`sample_tile.zig:117`).
- **Переносимый** — пресеты area-типов/флагов и сохранение правок через Persist
  (3.b, `registry_io`/`scene_io`).

Ценность: инженер редактирует разметку навмеша как в нормальном level-editor'е —
с Ctrl+Z, точным вводом, групповыми операциями и инкрементальной пересборкой, —
а не как в демо-визуализаторе «кликнул и молись».

## 2. Scope (in / out / YAGNI)

### Входит (in)

- **Команд-ориентированный undo/redo** (Command/EditOp + стек) для правок
  Scene-состояния: add/delete/move convex-volume, add/delete/move off-mesh,
  add/edit/remove area-type, add/remove poly-flag.
- **Snap** курсора правки: к вершинам/рёбрам геометрии (через уже существующий
  `InputGeom.raycastMesh` + ближайшая вершина), к XZ-сетке (шаг настраиваемый),
  к существующим точкам volume/концам connection.
- **Числовой ввод координат** точки/смещения (текстовые поля XYZ) как альтернатива
  пику мышью — для точного размещения и сдвига.
- **Мультивыделение** volumes/connections (rubber-band по XZ и Ctrl+click),
  **копирование** (с смещением) и **групповое перемещение** выделенного.
- **Пресеты** area-типов и наборов poly-флагов: сохранить текущий набор реестра как
  именованный пресет, применить пресет к Scene (через Persist `registry_io`).
- **Хоткей «rebuild only affected tiles»** — инкрементальная пересборка тайлов,
  пересекающих bbox правки, вместо полной `*.build()`.
- Все правки помечают `Scene.DirtyBits` (3.a) и сохраняются через Persist (3.b);
  инструменты регистрируются через UI-shell `tool_registry` (3.d); выделение
  рисуется через Render `highlight` (3.c).

### Не входит (out)

- Изменение faithful-ядра `src/*` (recast/detour). Используем только публичные
  API (`addTile`/`removeTile`/`calcTileLoc`/`buildTileMesh`).
- Редактирование самой геометрии меша (vertices/tris .obj) — F правит *разметку*
  (volumes/offmesh/area/flags), не исходный меш.
- Сериализация формата сцены и durable-запись — это **Persist (3.b)**; F только
  вызывает её API после правок.
- Новые визуальные схемы/легенды — это **Render (3.c)/кластер E**; F переиспользует
  `highlight`/`overlay`.
- Сетевой co-editing / multi-user — вне проекта.

### YAGNI (осознанно отложено)

- **Безграничный персистентный undo через journal-replay** (Persist этап 2,
  фундамент 3.b §«Этап 2»). На старте — **in-RAM** стек команд фиксированной
  глубины; персистентность undo — только если профиль правок потребует
  (фундамент явно: journal — YAGNI до подтверждённой потребности).
- **Поворот/масштаб** групп (affine) — на старте только translate + copy.
  Rotate/scale — отдельным этапом, если попросят.
- **Snap к произвольным CAD-примитивам** (дуги, сплайны) — только vertex/edge/grid.
- **Merge/split convex-volume** — вне старта.
- **Транзакции на несколько сцен** — H/headless (отдельный кластер).

### F1 — Undo/Redo для всех правок сцены (P0)

**Что.** Единый command-стек (`EditOp` tagged-union + undo/redo стеки) над
мутациями Scene-состояния: convex-volumes, off-mesh, area-типы, poly-флаги.

**Зачем.** Сейчас любая правка необратима (`deleteConvexVolume` /
`addOffMeshConnection` — без истории). Это самый частый и болезненный пробел
authoring-а: «удалил не тот volume / кликнул не туда» = ручное восстановление.
Фундамент §2/§4 прямо отводит undo/redo кластеру F поверх `Scene.DirtyBits`.

**UX.**
- **Ctrl+Z** — undo, **Ctrl+Y** / **Ctrl+Shift+Z** — redo. Гейтятся через
  UI-shell `input_gate` (не срабатывать, когда фокус в textfield —
  `ui_keyboard`, `main.zig:226`).
- Кнопки **Undo**/**Redo** в панели Tools (через UI-shell `panel`), disabled
  при пустом стеке; tooltip показывает имя следующей операции («Undo: Add Volume»).
- Опционально (P2): список последних N операций в Log-стиле.

**Данные.**
- `EditOp` — tagged-union, каждый вариант хранит **обратимое** описание правки
  (не снимок всей сцены): для `add_volume` — индекс/копия `ConvexVolume`; для
  `delete_volume` — копия удалённого + его прежний индекс; для `move_volume` —
  индекс + старые/новые `verts`; аналогично off-mesh (хранит все 6 полей
  параллельных массивов `InputGeom`, `input_geom.zig:32-39` — поэтому нужен
  *view*-снимок `OffMeshConn`, который фундамент 3.a уже планирует геттером
  `offMesh(i)`); для area/flags — старое/новое состояние записи реестра.
- Стек фиксированной глубины (кольцевой буфер, напр. 128 операций); владеет
  копиями данных правок (allocator — `Scene.alloc`).
- Каждая операция при apply/revert взводит соответствующий бит `Scene.DirtyBits`
  (geom/areas/flags/tiles) — это вход для F6 (инкрементальная пересборка) и
  Persist (3.b save).

**Зависимости.**
- **Scene (3.a)** — ●● команды мутируют под-модели через Scene-API (а не прямой
  `geom.volumes.orderedRemove`); `DirtyBits` — обязателен. **Блокер:** требует
  Scene-каркас (фундамент этап 2) и *view*-геттер off-mesh (`offMesh(i)`).
- **UI-shell (3.d)** — ●● хоткеи через `input_gate`, кнопки через `panel`.
- **Persist (3.b)** — ○ на старте undo in-RAM; durable-undo — YAGNI (см. §2).
- **src/\*** — нет прямых зависимостей (правки на demo-уровне).

### F2 — Snap к геометрии / сетке + числовой ввод координат (P0)

**Что.** Привязка точки правки (точка volume, конец off-mesh) к: (a) ближайшей
**вершине меша**, (b) **рёбру** меша, (c) узлу **XZ-сетки** (настраиваемый шаг),
(d) существующим точкам volume / концам connection. Плюс **числовой ввод XYZ**
(textfields) как альтернатива пику.

**Зачем.** Сейчас точка берётся только рейкастом меша (`pickPoint`,
`main.zig:471`) — «на глаз». Для повторяемой разметки (волюм ровно по краю
платформы, off-mesh точно между двумя площадками) нужна привязка и точный ввод.

**UX.**
- Тумблер **Snap mode** в панели инструмента (Off / Vertex / Edge / Grid),
  + слайдер **Grid step** (напр. 0.25…4.0). Активный snap-кандидат
  подсвечивается маркером (Render `overlay`/`highlight`, 3.c) до клика.
- Секция **Coords** с тремя textfields X/Y/Z (через `input_gate.claimKbd`):
  Enter ставит/двигает точку численно; поля заполняются текущим пиком, чтобы
  «доправить» координату.
- Hotkey-модификатор (напр. зажать **Ctrl**) — временно включить/выключить snap.

**Данные.**
- `SnapConfig { mode, grid_step }` — состояние инструмента (не Scene).
- `snapPoint(geom, raw_xyz, cfg) -> [3]f32` — чистая функция; vertex/edge через
  обход `geom.verts/tris` (как `raycastMesh`, `input_geom.zig:167`), grid —
  округление XZ к шагу. Возвращает также «тип привязки» для маркера.
- Числовой ввод — буферы textfield в инструменте, парсятся в `[3]f32`.

**Зависимости.**
- **Scene/InputGeom** — ● читает `geom.verts/tris` (snap-кандидаты); запись —
  через те же команды F1.
- **Render (3.c)** — ● маркер snap-кандидата (`overlay`/`highlight`).
- **UI-shell (3.d)** — ● textfields/гейты ввода (`input_gate`).
- **src/\*** — нет (геометрия читается на demo-уровне).

### F6 — Хоткей «пересобрать только затронутые тайлы» (P0)

**Что.** Инкрементальная пересборка: вместо полной `tile.build()` (все tw×th
тайлов) пересобрать **только тайлы, чей bbox пересекает bbox правки**.

**Зачем.** Полная пересборка на каждую правку (`main.zig:506-514`) — главный
тормоз authoring-а на больших тайловых мирах. `Scene.DirtyBits` (F1) уже знает
*что* и *где* изменилось — остаётся пересобрать минимум.

**UX.**
- Хоткей (напр. **B** или **Ctrl+B**) — «rebuild dirty tiles». Индикатор «N tiles
  dirty» в панели; красное уведомление при наличии dirty-тайлов (паттерн
  `main.zig:981` `! Navmesh rebuild needed`).
- Чекбокс **Incremental rebuild** (вкл по умолчанию для Tile); при выкл — старое
  поведение (полная `build()`). Solo — всегда полная (один тайл).

**Данные.**
- Множество «грязных» тайлов `dirty_tiles: AutoHashMap(struct{x,y}, void)`
  (уровень сэмпла Tile). Правка → bbox(volume/offmesh) → перечень (tx,ty) через
  `NavMesh.calcTileLoc` (`navmesh.zig:373`) по углам bbox → пометить.
- Инкрементальная пересборка: для каждого (tx,ty): `getTileAt`→`removeTile`
  (`navmesh.zig:537,1428`), затем `buildTileMesh(tx,ty,...)`
  (`sample_tile.zig:117`) → `addTile`. Это публичный путь, ядро не трогаем.

**Зависимости.**
- **src/detour/navmesh.zig** — ● `calcTileLoc`/`getTileAt`/`removeTile`/`addTile`
  (все уже публичны — проверено). **Только публичные API.**
- **sample_tile.zig** — ● нужен публичный `rebuildTile(tx,ty)` поверх приватного
  `buildTileMesh` (мелкая обёртка demo-уровня, не правка ядра).
- **Scene (3.a)** — ● `DirtyBits` как источник «что грязное».
- **Temp Obstacles** — частично: tilecache уже инкрементален по слоям
  (`sample_temp_obstacles.zig:250`); off-mesh/volumes там не основной кейс — для
  temp оставляем полную пересборку на старте (YAGNI).

### F3 — Мультивыделение / копирование / перемещение (P1)

**Что.** Выделить набор volumes и/или off-mesh connections, скопировать (со
смещением) и переместить группой.

**Зачем.** Поштучная работа (`onClick` по одному объекту) не масштабируется на
разметку из десятков зон. Групповые операции — стандарт level-editing.

**UX.**
- **Select-режим** инструмента редактирования: Ctrl+click — toggle объекта в
  выделении; rubber-band (drag по XZ) — выделить попавшие в прямоугольник
  (по центроиду volume / по концам connection). Выделенное подсвечивается
  (Render `highlight`, 3.c).
- **Ctrl+C / Ctrl+V** — копировать выделение в буфер и вставить со смещением
  (по умолчанию небольшой XZ-офсет; затем можно подвинуть).
- **Перемещение** — drag выделения по XZ (с активным snap из F2) либо численный
  офсет (Coords из F2). **Delete** — удалить выделение (через команды F1 — undo-able).

**Данные.**
- `Selection { volumes: BitSet/IndexList, offmesh: IndexList }` — индексы в
  `InputGeom`. **Внимание:** индексы инвалидируются при delete (orderedRemove
  сдвигает) — выделение хранить как индексы + ревалидировать после каждой
  команды, либо ввести стабильные id (см. ОВ-3).
- `Clipboard { volumes: []ConvexVolume, offmesh: []OffMeshConn }` — owned-копии.
- Групповые move/copy/delete — **компонуются как одна составная команда F1**
  (один undo откатывает всю группу).

**Зависимости.**
- **F1 (undo)** — ●● групповые операции = составные `EditOp`.
- **F2 (snap)** — ● при перемещении.
- **Render (3.c)** — ● `highlight` набора; rubber-band-прямоугольник overlay.
- **Scene/InputGeom** — ● индексы/копии объектов; **нужен стабильный id или
  ревалидация** (ОВ-3).
- **UI-shell (3.d)** — ● режим/хоткеи через `input_gate`.

### F4 — Шаблоны area-типов / наборов флагов (пресеты) (P1)

**Что.** Сохранить текущий набор area-типов и/или poly-флагов реестра как
именованный **пресет** и применять его к Scene.

**Зачем.** Проекты переиспользуют типовые наборы (ground/water/road/door…,
проектные флаги ladder/crouch). Сейчас они пересоздаются вручную в модальном
редакторе (`tool_convex.zig:219`) каждый раз.

**UX.**
- В панели area-редактора/Poly-Flags: **Save Preset** (имя) и дропдаун
  **Apply Preset**. Применение — это команда F1 (undo-able: заменяет/мерджит
  реестр, старое состояние в undo).
- Стратегия применения (radio): **Replace** (заменить реестр) / **Merge**
  (добавить недостающие) — по умолчанию Merge, чтобы не терять текущие.

**Данные.**
- Пресет = тот же бинарный формат, что `areas.reg`/`flags.reg` фундамента
  (Persist 3.b `registry_io`), но в каталоге пресетов (`presets/*.reg`).
  **Переиспользуем форматтер/парсер `registry_io`** — не вводим новый формат.
- Встроенные пресеты (seed из `area_types.ensureInit`, `area_types.zig:76-88`)
  как стартовый набор.

**Зависимости.**
- **Persist (3.b)** — ●● `registry_io` (areas.reg/flags.reg формат + writeAtomic).
  **Блокер:** требует фундамент этап 4 (Persist).
- **Scene (3.a)** — ● пресет применяется к `Scene.areas`/`Scene.flags` (после
  выноса из module-global, фундамент 3.a/этап 5) — иначе применяется к глобалям
  через переходные обёртки.
- **F1 (undo)** — ● apply как команда.

### F5 — Числовое редактирование свойств объекта (P2)

**Что.** Инспектор выделенного объекта: для volume — hmin/hmax/area/точки; для
off-mesh — start/end/radius/dir/area/flags — редактируемые поля.

**Зачем.** Уточнять параметры существующего объекта без удаления-пересоздания
(сейчас volume пересоздаётся целиком; off-mesh вообще не редактируется после
создания).

**UX.** Панель Properties для одиночного выделения (из F3): textfields/слайдеры;
изменение — команда F1; взводит DirtyBits → F6.

**Данные.** Reuse `ConvexVolume`/`OffMeshConn` поля; правка через Scene-API.

**Зависимости.** **F1**●●, **F3**● (выделение), **Scene**●. Низкий приоритет —
после P0/P1.

## 4. Архитектура

Конвенции — `.agent/project_structure.md`: snake_case, demo-уровень
(`demo/src/`), usize/owned на demo-уровне допустимо, faithful-ядро `src/*` не
трогаем. Все новые механизмы — поверх подсистем фундамента (ссылки, **не
передизайн**).

### Новые файлы `demo/src/`

```
demo/src/edit/
├── edit_op.zig        # EditOp tagged-union (add/del/move volume|offmesh,
│                      #   area add/edit/remove, flag add/remove, composite).
│                      #   apply(scene)/revert(scene); каждая взводит Scene.DirtyBits.
├── undo_stack.zig     # UndoStack: 2 кольцевых буфера (undo/redo) фикс-глубины,
│                      #   владеет копиями данных правок (Scene.alloc). push/undo/redo.
├── snap.zig           # SnapConfig + snapPoint(geom, raw, cfg) -> {pos, kind};
│                      #   vertex/edge (обход geom.verts/tris) / grid / object-point.
├── selection.zig      # Selection (индексы volumes/offmesh) + ревалидация после
│                      #   команд; rubber-band тест по XZ; Clipboard (owned-копии).
└── presets.zig        # area/flag-пресеты: load/save через persist/registry_io;
                       #   apply(scene, .replace|.merge) как EditOp.

demo/src/tool_edit.zig  # Объединённый authoring-инструмент (или расширение

```

### Точки интеграции (существующий код)

- **Команды вместо прямых мутаций.** `tool_convex.zig:103/119/195` и
  `tool_offmesh.zig:34` сейчас зовут `geom.addConvexVolume`/`deleteConvexVolume`/
  `addOffMeshConnection` напрямую. Заворачиваем в `EditOp` и проводим через
  `UndoStack.push(op.apply(scene))`. Сами `InputGeom`-методы остаются (команда —
  их вызывающий, плюс хранит обратную операцию).
- **Хоткеи undo/redo/rebuild** — в главном цикле рядом с обработкой клавиш
  (`main.zig` обработка key/Esc, gated `!ui_keyboard`), через UI-shell
  `input_gate` (3.d). Заменяет ad-hoc проверки.
- **Пересборка.** `main.zig:506-514` (`if dirty -> *.build()`) заменяется на:
  `if dirty_tiles непусто and incremental -> tile.rebuildDirtyTiles()` иначе
  полная `build()`. Нужна публичная `SampleTile.rebuildTile(tx,ty)` (обёртка над
  приватным `buildTileMesh`, `sample_tile.zig:117`) + `removeTile/addTile`
  (`navmesh.zig`).
- **DirtyBits.** Каждая команда взводит бит в `Scene.dirty` (фундамент 3.a поле
  `dirty: DirtyBits`); главный цикл читает их для F6 и для Persist-save.
- **Render выделения.** Через Render `highlight.zig`/`overlay.zig` (3.c) — те же
  примитивы, что у tester'а (`debugDrawNavMeshPoly`); F **не** заводит свой
  draw-слой.
- **Persist-save после правки.** После apply команды (или по таймеру/хоткею)
  Scene сохраняется через Persist `scene_io`/`registry_io` (3.b) — F вызывает,
  не реализует durable-запись.

### Что F **не** делает (границы)

- Не реализует Scene-агрегатор, DirtyBits-механику, off-mesh view-геттер — это
  **Scene (3.a)**. F их *потребитель*.
- Не реализует writeAtomic/checksum/registry-формат — это **Persist (3.b)**.
- Не реализует color-scheme/legend/highlight-примитивы — это **Render (3.c)**.
- Не реализует tool_registry/panel/input_gate — это **UI-shell (3.d)**.

## 5. Открытые вопросы / допущения к владельцу

> Явно — не угадываю. Каждый ОВ помечает допущение, на котором продолжаю.

1. **ОВ-1. Глубина/персистентность undo.** In-RAM стек фиксированной глубины
   (напр. 128) достаточно, или нужен durable-undo (journal-replay, Persist
   этап 2) с самого начала? *Допущение:* in-RAM фикс-глубины; durable —
   YAGNI до подтверждённой потребности (согласовано с фундамент §2 YAGNI).

2. **ОВ-2. Зависимость от незавершённого фундамента.** F1/F3/F4 фундаментально
   зависят от **Scene-каркаса (этап 2)**, **off-mesh view-геттера** и
   **Persist (этап 4)**, а F4 чисто блокируется Persist. Реализуем F2/F6 (минимум
   завязок) первыми, пока фундамент догоняет, или ждём фундамент целиком?
   *Допущение:* стартуем с F2+F6 (нужен только Scene-каркас и публичные
   navmesh-API, уже есть), F1/F3 — после Scene+off-mesh-геттера, F4 — после
   Persist.

3. **ОВ-3. Идентичность объектов для выделения/undo.** `InputGeom` хранит
   volumes/off-mesh как массивы; delete (`orderedRemove`) сдвигает индексы, ломая
   Selection и undo-ссылки. Вводим **стабильный per-object id** (поле id в
   `ConvexVolume` + off-mesh `off_id` уже есть, `input_geom.zig:38`) и адресуем по
   id, или ревалидируем индексы после каждой команды? *Допущение:* адресация по
   стабильному id (off-mesh `off_id` уже есть; для volume — добавить id на
   demo-уровне), это устойчивее и нужно для durable-undo позже. **Требует
   согласования со Scene (3.a), т.к. меняет модель.**

4. **ОВ-4. Объём snap.** Достаточно vertex/edge/grid/object-point, или нужен
   snap к «поверхности под курсором с заданной высотой» / к рёбрам уже собранного
   navmesh-полигона? *Допущение:* vertex/edge/grid/object-point на старте;
   navmesh-edge snap — позже (зависит от Render/выделения полигонов).

5. **ОВ-5. Хоткеи.** Предлагаю Ctrl+Z/Ctrl+Y(+Ctrl+Shift+Z), Ctrl+C/V, Delete,
   B (rebuild dirty), Ctrl — временный snap-toggle. Конфликтуют ли с
   существующими (C=cull, P=cam-print, V=voxel-cycle, `FEATURES.md §6`)?
   *Допущение:* edit-хоткеи активны только в edit-инструменте и gated
   `!ui_keyboard`; одиночные C/P/V остаются глобальными — беру **B** (свободна) и
   **Ctrl-комбинации** (не конфликтуют с одиночными).

6. **ОВ-6. Granularity dirty-тайлов для F6.** Помечать тайлы по bbox правки (с
   учётом `border_size`-расширения, `sample_tile.zig:130`), или консервативно
   расширять на ±1 тайл (off-mesh/volume у границы влияет на соседей)?
   *Допущение:* bbox + border-расширение углов через `calcTileLoc`; при сомнении
   расширяю на соседние тайлы (корректность важнее минимизма).

7. **ОВ-7. Temp Obstacles в F6.** Инкрементальный rebuild для tilecache-сэмпла
   (off-mesh/volume там редки) — делаем или оставляем полную пересборку?
   *Допущение:* для Temp — полная пересборка на старте (YAGNI); tilecache и так
   инкрементален по препятствиям.

## 6. Риски

- **R-F1 — Зависимость от ещё-не-готового фундамента.** F1/F3/F4 не собрать без
  Scene-каркаса (этап 2), off-mesh-геттера и Persist (этап 4). *Митигация:*
  порядок этапов (§7) ведёт F2/F6 вперёд (минимальные завязки), остальное — по
  мере готовности фундамента; явные блокеры в §5 ОВ-2.

- **R-F2 — Инвалидация индексов (orderedRemove).** Selection/undo по индексам
  ломаются при delete. *Митигация:* адресация по стабильному id (ОВ-3); до
  решения — ревалидация после каждой команды (дорого, но корректно).

- **R-F3 — Корректность инкрементальной пересборки (F6).** Неполный набор
  dirty-тайлов → дыры/рассинхрон с полной сборкой; правка у границы тайла влияет
  на соседей через `border_size`. *Митигация:* консервативное расширение
  (ОВ-6); **регрессионный тест** (`test/integration/`): инкрементальная сборка ==
  полная `build()` побайтно на наборе правок.

- **R-F4 — Раздувание `main.zig`.** Edit-хоткеи/диспетч в монолите. *Митигация:*
  всё через UI-shell `input_gate`/`tool_registry` (3.d) — фундамент R5 уже об
  этом; F не добавляет инлайн-логику в `main.zig`, только тонкие вызовы.

- **R-F5 — Память undo-стека.** Команды владеют копиями (volume verts, off-mesh
  6 полей); глубокий ст稿 на больших правках. *Митигация:* фикс-глубина (кольцо),
  компактные обратные описания (не снимки всей сцены), освобождение при вытеснении.

- **R-F6 — Module-global реестры (area/flags).** Пока area_types/poly_flags —
  глобали (`area_types.zig:50`, `poly_flags.zig:40`), undo/пресеты на них хрупки
  (нет точки истины, конфликт с будущими инстансами Scene). *Митигация:* идти
  через Scene-API/переходные обёртки (фундамент 3.a решение Q2); полноценно —
  после выноса в инстанс (фундамент этап 5).

- **R-F7 — Конфликт хоткеев / гейтов ввода.** Edit-комбо vs камера/глобальные.
  *Митигация:* активны только в edit-инструменте и `!ui_keyboard`; набор сверен с
  `FEATURES.md §6` (ОВ-5).

## 7. Этапы реализации (порядок)

Порядок ведёт фичи с минимальными завязками на незавершённый фундамент вперёд;
блокирующие зависимости (§5 ОВ-2) отмечены.

1. **F6 — инкрементальная пересборка (P0).** `SampleTile.rebuildTile(tx,ty)`
   (обёртка над `buildTileMesh`) + `dirty_tiles`-множество + переключение
   `main.zig:506-514` на инкрементальный путь. Хоткей B. **Регрессионный тест:
   инкрементальная == полная.** *Завязки:* только публичные navmesh-API (есть) +
   Scene.DirtyBits (этап 2 фундамента — минимально).
   *Разблокирует:* быстрый authoring-цикл для всего остального.

2. **F2 — snap + числовой ввод (P0).** `snap.zig` (vertex/edge/grid/object) +
   Coords-textfields + маркер snap-кандидата (Render overlay). *Завязки:* читает
   `geom` (есть), Render `overlay`/`highlight` (3.c этап 3), UI-shell
   `input_gate` (3.d этап 1).

3. **F1 — undo/redo (P0).** `edit_op.zig` + `undo_stack.zig`; заворачивание
   существующих мутаций convex/offmesh в команды; хоткеи Ctrl+Z/Y; кнопки.
   *Завязки (блокер):* Scene-каркас + `DirtyBits` + off-mesh view-геттер
   (фундамент 3.a/этап 2). Решить ОВ-3 (стабильные id) **до** реализации.

4. **F3 — мультивыделение / копирование / перемещение (P1).** `selection.zig`
   (rubber-band, clipboard) + групповые составные команды (на F1). *Завязки:* F1,
   F2, Render `highlight` (3.c), стабильные id (ОВ-3).

5. **F4 — пресеты area/flags (P1).** `presets.zig` поверх Persist `registry_io`;
   apply как команда. *Завязки (блокер):* Persist этап 4 (фундамент) + Scene
   areas/flags (этап 2/5).

6. **F5 — числовое редактирование свойств (P2).** Инспектор выделенного объекта.
   *Завязки:* F1, F3, Scene. Полировка после P0/P1.

**Критический путь:** **1 → 2 → 3 → 4** (F6 и F2 идут параллельно фундаменту;
F1 — как только готов Scene-каркас + off-mesh-геттер). F4 — после Persist. F5 —
последним.
