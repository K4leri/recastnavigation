## 2. Порядок реализации фундамента (критический путь)

Из `foundation §7` (с учётом решений владельца: per-tile сразу, XXH3, архив-экспорт):

1. **U-shell минимум** — `input_gate.zig` (формализация `ui_mouse`/`ui_keyboard`) +
   `tool_registry.zig` поверх `SampleTool` vtable. Разгружает `main.zig`.
2. **Scene-каркас** — `scene.zig` оборачивает `InputGeom`/`area_types`/`poly_flags`/
   `CommonSettings` + `SceneMeta`(uuid/версия) + `DirtyBits`. **Здесь же решить
   стабильные per-object id** (нужно F/I — см. §4 D2). Module-global — через
   переходные обёртки.
3. **Render/overlay** — `color_scheme`+`highlight`+`overlay`, затем `legend`+
   `isolation`. Разблокирует E (осн.), A/B/C/G/J.
4. **Persist (per-tile)** — `write_atomic`+`checksum`(XXH3)+header; директория
   `.recastscene/` (`registry_io`/`scene_io`/`tile_store`/`manifest`) +
   `packArchive`/`unpackArchive`. Разблокирует I/D/G/F/H.
5. **Полный вынос module-global** `area_types`/`poly_flags` в `Scene`. Разблокирует H.
6. **Persist этап 2 (журнал)** — YAGNI до подтверждённой потребности.

## 3. Волны реализации кластеров

**Волна 0 — параллельный «pure-logic» трек (НЕ ждёт GUI-фундамент).**
GUI-независимая, тестируемая логика — можно начать сразу, в `test/integration/`:
- A: `why_no_path.zig`, `components.zig` (машина вердиктов + связность).
- B: сбор счётчиков пост-обходом `hf/chf/cset/pmesh/dmesh`.
- C: `build_profile.zig` (тайминги обёрткой `nanoTime` вокруг стадий `sample_solo`).
- G: `navmesh_verify.zig` (вынос `countFreeLinks`/`linksToTile` из теста) + `connectedComponents` (обобщение `floodNavmesh` из `tool_prune`).
- D: импортёры геометрии (glTF/PLY/STL) + метрики-дайджест.
- J: `why_stuck.zig` классификатор по полям `CrowdAgent`.
- I: эмпирический тест детерминизма ядра (нужен G/I/J).

**Волна 1 — после R+U (шаги 1–3 фундамента):** E (основной потребитель R), затем
GUI-части A/B/C, P0 J (why-stuck + tilecache-timeline).

**Волна 2 — после Persist (шаг 4):** I (save/load+bundle), D (scene/CLI),
G (golden), F (undo/snap/rebuild-tiles + пресеты), I/O-части J (record/replay).

**Волна 3 — после выноса module-global (шаг 5):** H (headless+config+batch),
параллельные batch-части C/D/G.

## 4. Сквозные решения для владельца (затрагивают несколько кластеров)

> Эти вопросы повторяются в спеках; решить ОДИН раз, до старта затронутых кластеров.

- **D1 — Политика касания ядра `src/*` (read-only геттеры).** Несколько кластеров
  хотят добавить в ядро НЕинвазивные read-only геттеры/итераторы для точных метрик:
  C-Q3 (node-pool «узлов/запрос», число регионов), B-Q5 (счётчики регионов),
  J-Q1 (`pendingUpdateCount` + итератор затронутых тайлов), A-Q1 (опциональный
  `h_scale` для Dijkstra-плеера). Все — *аддитивные, не меняют faithful-алгоритмы*.
  **Вопрос:** разрешить такие геттеры (в отдельной `*_introspect.zig`-секции,
  явно помеченной как demo-support), или жёстко «ядро не трогаем» (тогда метрики
  приблизительные/теневые)? *Рекомендация:* разрешить аддитивные read-only геттеры.
- **D2 — Стабильные per-object id в Scene.** F (selection/undo) и I требуют
  стабильных id для volumes (у off-mesh `off_id` уже есть). Индексы ломаются на
  `orderedRemove`. **Решать на шаге 2 фундамента** (меняет Scene-модель).
  *Рекомендация:* добавить `id` в convex volume в Scene-обёртке.
- **D3 — Детерминизм ядра.** G3 (golden), I (verify), J (replay) зависят от
  байт-детерминизма построения. Спек I проверил статикой: ядро детерминировано
  (RNG нет в навмеш-пути). **Действие:** ранний integration-тест «двойной build →
  идентичный дайджест» как контракт. *Рекомендация:* добавить в Волну 0.
- **D4 — Граница render-фундамент (3.c) vs кластеры E/B/A/C.** По решению Q7
  фундамента: обобщённые механизмы (схемы/легенды/изоляция/выделение) — в 3.c;
  конкретные визуализации — в кластерах. E-Q6/B-граница/A-Q3/C — следуют этому.
- **D5 — glTF-парсер (D-Q2).** Свой минимальный (`std.json`+accessors) vs внешняя
  Zig-зависимость. *Кластер-локально, решить при старте D.*

## 5. Рекомендованный первый шаг

Критический путь — фундамент. Рекомендую стартовать:
1. **Параллельно:** Волна-0 pure-logic трек (A/B/C/G/D/J/I-логика в `test/integration/`)
   — даёт ценность и тесты без ожидания GUI.
2. **Последовательно:** фундамент шаг 1 (U-shell) → шаг 2 (Scene-каркас + D2) →
   шаг 3 (Render) → шаг 4 (Persist).

По каждому выбранному шагу — отдельный цикл `writing-plans` → реализация.
