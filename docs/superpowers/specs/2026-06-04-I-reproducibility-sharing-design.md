## 1. Цель и ценность для developer-user

Сегодня (см. `FEATURES.md` §11) **состояние сцены не персистится**: convex-объёмы,
off-mesh-связи, реестры area/flags живут только в RAM сессии. На диск пишется
лишь *собранный* navmesh (MSET, `demo/src/navmesh_io.zig`) — но не исходная
разметка, по которой он собран, не параметры сборки, не тестовый запрос. Из-за
этого developer-user не может:

- **вернуться** к рабочей сцене после перезапуска (всё перенастраивать заново);
- **переслать** коллеге воспроизводимый кейс «вот геометрия + вот эти настройки +
  вот этот запрос даёт неверный путь» — приходится словесно описывать;
- **доказать** что баг воспроизводится: между запусками не гарантирован
  байт-в-байт идентичный навмеш (нет явного repro-контракта/seed/версий).

Кластер I закрывает три связанных сценария поверх фундамента:

1. **Save/Load полного state сцены** — точка возврата к рабочему состоянию мира
   (Sc через P, durable-контейнер `.recastscene/`).
2. **Bug-report bundle** — *один* файл-архив (геометрия + правки + параметры +
   запрос + ожидаемый/полученный результат), который пересылается и
   разворачивается в идентичную сцену (P.`packArchive`/`unpackArchive`, решение
   фундамента Q3).
3. **Детерминизм** — явный контракт «фиксированные входы → байт-в-байт идентичный
   навмеш между запусками/машинами», с фиксацией версий ядра/форматов и
   verify-режимом.

Ценность: «поделись сценой» и «воспроизведи мой баг» становятся **встроенными
one-click сценариями**, а не ad-hoc пересылкой `.obj` + скриншота + словесного
описания. Это фундамент для кластеров D (импорт/экспорт), G (золотые
снимки/регрессии), F (save после правки).

## 2. Scope (in / out / YAGNI)

### Входит (in)

- **UI-слой** Save/Load сцены: команды в shell (U), диалог пути, статус в Log
  (`build_context.zig`). Сама сериализация — это P (фундамент), кластер I её
  *вызывает и оркеструет*, не переопределяет.
- **Bundle**: формат-обёртка `.recastbundle` (один файл) = `packArchive` от
  директории-сцены `.recastscene/` + добавочная секция `repro/` (манифест
  запроса + результат + repro-метаданные). Export/Import как demo-команды.
- **Repro-манифест запроса** (`repro/query.json` или бинарь): какой инструмент,
  какие точки start/end, include/exclude-флаги, какой режим (Follow/Straight/
  Sliced/Raycast/...), и **снимок результата** (polys/waypoints/raycast-hit) на
  момент экспорта — как «ожидаемое» для сравнения при импорте.
- **Repro-контракт детерминизма** (demo-уровень): фиксация и запись в манифест
  версий — `core_version` (тег порта), `mset_version` (`navmesh_io.zig` VERSION),
  версий наших edits/tiles-форматов (P), хэша геометрии (XXH3 из P.checksum), всех
  `CommonSettings`. Verify-режим: пересобрать → сравнить контрольный хэш навмеша.
- **CLI-вход**: запуск `recast_demo <bundle>` / drag-bundle-onto-exe → авто-импорт
  и переход в воспроизведённое состояние (расширение существующего парсинга
  аргументов `main.zig`).

### Не входит (out / YAGNI)

- **Реализация durable-записи/checksum/манифеста/archive** — это **фундамент P**
  (`demo/src/persist/*`, спек 3.b). Кластер I их **потребляет**, не пишет.
  `packArchive`/`unpackArchive`, `scene_io`, `manifest`, `tile_store`,
  `write_atomic`, `checksum` — поставляет фундамент (решения Q3/Q4/Q5).
- Сетевой шаринг/облако/загрузка по URL, аккаунты, серверный реестр багов
  (фундамент §2 явно out). Bundle — это локальный файл, пересылается «руками».
- Undo/redo, история правок (это кластер F поверх Sc.DirtyBits).
- Полноценный diff навмешей/визуальный 3-way merge (кластер G — золотые снимки;
  I даёт только бинарный verify-хэш «совпало/нет», не покабельный diff).
- Кроссплатформенный байт-в-байт детерминизм между **разными CPU-арке**
  (x86 vs ARM, разный FP-rounding). Цель — детерминизм на **одной арке/тулчейне**
  (Win x86_64 / Linux x86_64, тот же тег порта). См. риск R3.
- Шифрование/подпись bundle (фундамент §2: XXH3 — детект bitrot, не защита).
- Сжатие bundle сверх того, что даёт `packArchive` (если P не сжимает — bundle
  тоже не сжимает; YAGNI до подтверждённого размера).

### Зависимость от ещё-не-реализованного фундамента (явно)

На момент написания (`Glob demo/src/persist/**` → пусто, `scene.zig` → нет)
**ни Scene (Sc), ни Persist (P), ни shell (U) ещё не реализованы** — это
дизайн-спеки. Кластер I **блокирован** на:

- **Sc**: `Scene` агрегат + `SceneSnapshot` + `SceneMeta{uuid, версия формата,
  ссылка на geometry}` + `DirtyBits` (фундамент 3.a). Без `SceneMeta.uuid` нет
  идентичности сцены; без `SceneSnapshot` нечего сериализовать атомарно.
- **P**: `scene_io.packArchive/unpackArchive` (директория ⇄ один файл, Q3),
  `write_atomic`, `checksum` (XXH3, Q5), `manifest`, `registry_io`, `tile_store`
  (фундамент 3.b, этап 1 — Q4).
- **U**: `tool_registry`/`panel` для команд Save/Load/Export/Import без правки
  монолита `main.zig` (фундамент 3.d).

I не должен начинаться раньше фундамент-этапов 1→2→4 (см. фундамент §7).

## 3. Фичи (приоритезированы)

Приоритеты: **P0** = ядро кластера (без него I не существует), **P1** = делает
кластер ценным, **P2** = удобство.

### I-1 Save/Load полного state сцены (P0)

**Что.** Команды «Save Scene» / «Load Scene» в demo: сериализуют/десериализуют
*весь* редактируемый state (геометрия-ссылка + convex volumes + off-mesh +
area-реестр + flag-реестр + `CommonSettings` + meta) в durable-контейнер
`.recastscene/` (директория, фундамент 3.b) и обратно. Опционально — также
запечённые тайлы навмеша (`tiles/`), чтобы Load не требовал пересборки.

**Зачем.** Точка возврата к рабочему состоянию мира между сессиями; основа для
bundle (I-2). Закрывает `FEATURES.md` §11 «нет персистентности состояния сцены».

**UX.**
- Команды через U-shell (новая панель/пункты, не правка монолита). Save → диалог
  имени `<name>.recastscene`; Load → выбор существующего контейнера.
- Save **атомарен и целостен** — гарантирует P (atomic-rename manifest последним,
  per-tile + XXH3). Кластер I только показывает прогресс/итог в окне Log
  (`bctx.context().log(...)`, паттерн `main.zig:774`).
- Дефолтный каталог — рядом с exe/cwd (как сканирование ассетов сейчас,
  `FEATURES.md` §11 «`test_data` рядом с exe/cwd»).
- После Load: инструменты сбрасываются и пересоздают query (как при Load MSET
  сейчас, `FEATURES.md` §9 «Загрузка заменяет текущий navmesh и сбрасывает
  инструменты»); при отсутствии `tiles/` — авто-Build (или подсказка
  `! rebuild needed`, как `FEATURES.md` §3).

**Данные.** `SceneSnapshot` (Sc 3.a) → P пишет: `scene.gset` (геометрия as-is),
`edits/{areas.reg, flags.reg, volumes.bin, offmesh.bin}`, опц. `tiles/*.tile`,
`manifest`. Кластер I добавляет только `meta`-вызов (uuid/версия из
`SceneMeta`). Маппинг текущих структур → формат **полностью** в фундаменте 3.b
(таблица «edits/»); I не переопределяет байт-лейаут.

**Зависимости.** Sc (`Scene`/`SceneSnapshot`/`SceneMeta`) ●●; P
(`scene_io`/`manifest`/`tile_store`/`registry_io`/`write_atomic`) ●●; U (команды)
●; от ядра `src/*` — нет прямых (через Sc). Текущие `input_geom.zig`
(`volumes`/`off_*` параллельные массивы, стр. 30-52), `area_types.zig`,
`poly_flags.zig` — читаются через Sc-обёртки (фундамент 3.a), не напрямую.

### I-2 Bug-report bundle (single-file архив) (P0)

**Что.** «Export Bundle» — упаковать текущую сцену **+ активный repro-запрос**
(I-4) в *один* файл `<name>.recastbundle` для пересылки. «Import Bundle» —
развернуть его в идентичную сцену и (опц.) в воспроизведённое состояние запроса.
Bundle = `packArchive(.recastscene/)` (P, решение Q3) с **добавочной секцией**
`repro/` поверх содержимого сцены.

**Зачем.** Главный сценарий шаринга: «вот мой баг одним файлом». Получатель
открывает bundle и видит ровно ту же геометрию/правки/настройки/запрос.

**UX.**
- «Export Bundle» (U-команда) → один файл; имя по `SceneMeta` (`<scene>-<uuid8>.recastbundle`).
- «Import Bundle» → распаковать во временную/рабочую директорию-сцену (P.unpack),
  загрузить (I-1), затем восстановить запрос (I-4) и **показать verify-итог**
  (см. I-3): «навмеш совпал с эталоном bundle ✓ / РАЗОШЁЛСЯ ✗ (см. Log)».
- Drag bundle на окно/exe или CLI-arg → то же (I-5).

**Данные.** Bundle-контейнер (логически):
```
<name>.recastbundle           # один файл = packArchive контейнера-сцены + секция repro/
  ├─ (всё содержимое .recastscene/: scene.gset, *.obj, edits/, tiles/, manifest)
  └─ repro/
     ├─ query.bin             # манифест запроса (I-4): tool, точки, флаги, режим
     ├─ result.bin            # снимок результата на момент export (эталон для verify)
     └─ repro.meta            # core_version, mset_version, формат-версии, geom XXH3,
                              #   полный CommonSettings (I-3 repro-контракт)
```
- **Формат-обёртка bundle** = тот же archive-формат, что даёт P.`packArchive`
  (НЕ изобретаем zip). Секция `repro/` — это просто доп. файлы внутри
  директории-сцены под `repro/`, попадающие в `packArchive` естественно. То есть
  **bundle ≡ packArchive сцены, у которой есть подкаталог `repro/`**. Не нужен
  отдельный контейнерный формат — нужен лишь уговор «`repro/` внутри сцены».
- `*.obj`: **встраивается в bundle обязательно** (иначе у получателя нет
  геометрии — `.gset` ссылается строкой `f %s` на внешний `.obj`, фундамент 3.b).
  Для save-сцены (I-1) `.obj` может оставаться внешним; для bundle — копируется
  внутрь. Это ключевое отличие bundle от scene (см. открытый вопрос Q-I3).

**Зависимости.** P (`packArchive`/`unpackArchive` — Q3) ●●; I-1 (сцена) ●●;
I-4 (repro-запрос) ●●; I-3 (repro.meta) ●●; U ●.

### I-3 Детерминизм навмеша (repro-контракт) (P1)

**Что.** Контракт + verify-механизм: «при тех же входах (геометрия + settings +
версия ядра) Build даёт **байт-в-байт идентичный** навмеш». Реализуется как (а)
запись repro-метаданных в bundle/scene; (б) demo-режим `verify`: пересобрать
из bundle и сравнить XXH3 итогового навмеша с эталоном из `repro.meta`.

**Зачем.** Без этого «воспроизведи мой баг» не гарантирован: получатель может
собрать *другой* навмеш и не увидеть проблему. Даёт объективный критерий
«воспроизвелось / нет».

**UX.**
- При Export: посчитать XXH3 (P.checksum) итогового навмеша (конкатенация
  tile-blob'ов в детерминированном порядке — по `(tx,ty,layer)`), записать в
  `repro.meta` вместе с версиями.
- При Import: после авто-Build пересчитать тот же XXH3 → сравнить. Итог в Log:
  `repro OK` / `repro MISMATCH (core_version A vs B / settings drift / ...)` с
  детализацией, *какое* поле разошлось (версия ядра ≠ / settings ≠ / geom-хэш ≠).
- CLI `--verify=<bundle>` (headless-дружелюбно, кластер H): exit 0 при совпадении,
  ≠0 при mismatch — для CI/регрессий (кластер G).

**Данные.** `repro.meta`:
```
core_version:   строка-тег порта (git-tag/коммит, напр. "0.1.1"; см. release-теги)
mset_version:   u32 (navmesh_io.zig VERSION=1)
edits_versions: версии areas/flags/volumes/offmesh форматов (P)
settings:       полный снимок CommonSettings (cell_size..max_sample_error, FEATURES §3)
geom_hash:      XXH3 геометрии (verts+tris, нормализованный) — P.checksum
navmesh_hash:   XXH3 итогового навмеша (эталон verify)
partition:      watershed|monotone|layers (NB: см. ниже про текущую заглушку)
```

**Анализ источников (не)детерминизма (проверено по `src/*`).**
- **Ядро `src/*` детерминировано**: проверка показала — нет системного RNG/
  `nanoTimestamp`/`@intFromPtr`-зависимостей в пути сборки. Совпадения «rand» в
  `src/recast/region.zig` — это подстрока `@rem` (стр. 710/717), не rng;
  `src/detour_crowd/obstacle_avoidance.zig` — детерминированное адаптивное
  сэмплирование (без rng). Порядок обхода — фиксирован алгоритмами Recast.
  Вывод: **при одинаковых float-входах и той же арке навмеш детерминирован.**
- **Недетерминизм только на demo-уровне и НЕ влияет на байты навмеша:**
  `io_util.PerfTimer` (стр. 96-111) — лишь замер времени (в Log/`time_ms`), в
  навмеш не попадает; idle/FPS. Их в `navmesh_hash` НЕ включаем.
- **Открытая дыра faithful-конвейера:** `FEATURES.md` §11 — Partitioning/Filtering
  это «только UI-чекбоксы», конвейер сэмплов **жёстко** watershed + 3 фильтра
  (значения не разветвляют сборку). Значит `partition` в repro.meta сейчас
  **информативен, но не управляет сборкой** — фиксируем фактическое поведение
  (watershed) и документируем; verify это учитывает (не обещаем repro по полю,
  которое сборка игнорирует). Если кластер B/F позже подключит разветвление —
  repro-контракт расширяется (НЕ ломая формат: версия `repro.meta`).

**Зависимости.** P (`checksum` XXH3) ●●; Sc (`CommonSettings`, `SceneMeta`) ●;
ядро `src/*` — **только чтение/верификация, НЕ изменение** (faithful незыблем);
сэмплы (`sample_*.zig` Build) — вызов пересборки.

### I-4 Repro-манифест запроса (query+result) (P1)

**Что.** Сериализовать «активный запрос» инструмента Test Navmesh
(`tool_navmesh_tester.zig`) + его результат: точки start/end, include/exclude-
флаги, режим (Follow/Straight/Sliced/DistanceToWall/Raycast/FindPolys*), и
снимок результата (кол-во polys/waypoints, straight-path точки, raycast-hit) —
аналогично тому, как `testcase.zig` уже хранит кешированный результат
(`Test{spos,epos,include,exclude,nstraight,straight,ray_hit,...}`, стр. 15-28).

**Зачем.** Bundle бесполезен без «вот ЭТОТ запрос ломается». Снимок результата =
«ожидаемое» (на машине-отправителе) для сравнения при импорте — суть bug-report.

**UX.**
- «Export Bundle» подхватывает текущее состояние тестера (если активен), пишет в
  `repro/query.bin` + `repro/result.bin`.
- При Import — восстановить точки/флаги/режим тестера, пересчитать запрос на
  загруженном навмеше, **показать diff** с эталоном result.bin: «N полигонов
  совпало / путь короче на K точек / raycast hit сместился» (текстом в Log;
  визуальный diff — кластер A/G).

**Данные.** Переиспользуем формат, **изоморфный** `testcase.zig` (расширенный):
- общая часть совпадает с TestCase-строками `pf`/`rc` (`testcase.zig:163-179`) —
  можно экспортировать repro-запрос **также** в человекочитаемый `TestCases/*.txt`
  (бонус: совместимость с существующим окном Test Cases, `FEATURES.md` §8);
- бинарная `query.bin`/`result.bin` — для режимов, которых нет в txt-формате
  (Sliced/DistanceToWall/FindPolys*), с header magic/version/XXH3 (P.checksum).

**Зависимости.** `tool_navmesh_tester.zig` (источник состояния запроса) ●●;
`testcase.zig` (образец сериализации запроса/результата, переиспользуем парсер
`pf`/`rc`) ●; P (`checksum`, header) ●; ядро `src/detour` (`NavMeshQuery`) —
только чтение результата.

### I-5 CLI bundle import (drag-onto-exe / arg) (P2)

**Что.** Запуск `recast_demo <path.recastbundle>` или drag bundle на exe →
авто-Import (I-2) + Verify (I-3) на старте, переход в воспроизведённое состояние.

**Зачем.** «Кинул файл — увидел баг» без ручной навигации по меню. Удобство
шаринга; мост к headless (кластер H) и CI (кластер G).

**UX.** Расширение существующего парсинга аргументов (`main.zig`, ср.
`FEATURES.md` §10: `--cam`/`--draw`/`--bench`). Новые: позиционный `<bundle>`,
`--verify=<bundle>` (headless exit-code), `--export-bundle=<out>` (собрать bundle
из текущих ассетов скриптом — для batch).

**Зависимости.** I-2/I-3 ●●; U (диспетч) ○; `main.zig` arg-parsing (расширить).

## 4. Архитектура

### Новые файлы (demo-уровень, snake_case; `.agent/project_structure.md`)

```
demo/src/repro/
├── scene_session.zig   # ОРКЕСТРАТОР Save/Load сцены (I-1). Тонкий слой над P:
│                       #   собирает SceneSnapshot (Sc) -> зовёт scene_io/manifest (P);
│                       #   обратно: unpack -> Scene + сброс инструментов + (re)build.
│                       #   НЕ содержит байт-лейаута (он в P/3.b).
├── bundle.zig          # I-2: exportBundle(scene, repro)/importBundle(path).
│                       #   = P.packArchive поверх .recastscene/ с подкаталогом repro/.
│                       #   Добавляет/извлекает секцию repro/ (query/result/meta).
├── repro_manifest.zig  # I-3: repro.meta read/write (core_version/mset_version/
│                       #   settings/geom_hash/navmesh_hash/partition); header+XXH3 (P).
│                       #   navmeshHash(mesh): детерминированная конкатенация tile-blob
│                       #   по (tx,ty,layer) -> P.checksum.XXH3.
├── query_record.zig    # I-4: сериализация состояния tool_navmesh_tester (+ результата).
│                       #   reuse парсера pf/rc из testcase.zig; бинарь для Sliced/etc.
└── cli.zig             # I-5: разбор <bundle>/--verify/--export-bundle; ветка headless.
```

### Точки интеграции (что трогаем, минимально)

- **U-shell (3.d)** — регистрируем команды Save/Load Scene, Export/Import Bundle
  как пункты панели/инструмент через `tool_registry`/`panel` (фундамент 3.d).
  **Без правки монолита `main.zig`** сверх подключения (это и есть смысл shell).
- **Sc (3.a)** — читаем `SceneSnapshot`/`SceneMeta`; **не добавляем полей в Scene**
  сверх того, что фундамент 3.a уже закладывает (`meta: SceneMeta{uuid,версия,
  ссылка на geometry}`, `dirty: DirtyBits`). Если I нужен `navmesh_hash`-кэш —
  держим его в `repro/`, не в Sc.
- **P (3.b)** — **только вызываем** `packArchive`/`unpackArchive`/`scene_io`/
  `manifest`/`tile_store`/`registry_io`/`write_atomic`/`checksum`. Если нужного
  API нет — это **запрос к фундаменту**, не реализация в I (см. открытые вопросы).
- **`main.zig`** — расширить arg-parsing (I-5): позиционный bundle + `--verify` +
  `--export-bundle`; ветку headless `--verify` (без окна, exit-code).
- **`tool_navmesh_tester.zig`** — добавить геттер «снять текущее состояние запроса»
  и сеттер «восстановить» (для I-4). Маленький read/write API на инструменте,
  faithful-логику запроса не трогаем.

### Что НЕ трогаем

- `src/*` (faithful-ядро) — **ноль изменений**. Детерминизм (I-3) — это
  верификация существующего поведения, не правка алгоритмов.
- Байт-лейаут durable-форматов, `.gset`-семантика, archive-формат — **владение P**
  (фундамент 3.b). I не переопределяет (риск рассинхрона форматов).
- `navmesh_io.zig` MSET — переиспользуем как образец/как tile-blob (фундамент 3.b
  уже планирует MSET-blob в `tile_store`); не дублируем.

### Поток данных (Export Bundle, целиком)

```
Scene (Sc) ──snapshot──> SceneSnapshot
                              │
        scene_session.save ───┼──> P.scene_io/registry_io/tile_store/manifest
                              │       └─> .recastscene/ (atomic, XXH3)  [I-1]
query_record(tool_tester) ────┼──> .recastscene/repro/query.bin,result.bin  [I-4]
repro_manifest(versions,hash)─┴──> .recastscene/repro/repro.meta            [I-3]
                                          │
                          bundle.export ──┴──> P.packArchive ──> <name>.recastbundle [I-2]
```

## 5. Открытые вопросы / допущения к владельцу

Фундамент уже зафиксировал Q1-Q7 (директория `.recastscene/` + обязательный
`packArchive`, этап 1 per-tile, XXH3, единый `CommonSettings`). Ниже — вопросы,
**специфичные для кластера I**, которые фундамент не закрывает:

1. **Q-I1 (репро-версия = git-тег?).** `core_version` для детерминизма —
   привязывать к git-тегу/коммиту порта (как release-теги `0.1.1`)? Это требует
   прокидывания версии в бинарь на build-time (build.zig `-D`/`@embedFile` или
   build-options). *Допущение:* да — build-option `core_version` из build.zig.zon
   `version`; mismatch версии при Import = warning, не блок (баг может
   воспроизводиться и на другой версии).

2. **Q-I2 (verify по `partition` — заглушка).** Faithful-конвейер сейчас
   **игнорирует** Partitioning/Filtering-чекбоксы (`FEATURES.md` §11, всегда
   watershed+3 фильтра). Фиксировать `partition=watershed` как факт и НЕ обещать
   repro по этому полю, пока B/F не подключат разветвление? *Допущение:* да —
   пишем фактическое поведение, помечаем поле `informational` до подключения.

3. **Q-I3 (geometry: внешняя ссылка vs встроенная).** Для **scene** (I-1) `.obj`
   остаётся внешним (как `.gset` сейчас, `f %s`-ссылка), для **bundle** (I-2)
   `.obj` встраивается обязательно (получатель не имеет файла). Верно ли это
   разделение, или scene тоже всегда встраивает геометрию (проще, но дублирует
   мегабайты ассетов на каждый save)? *Допущение:* scene — ссылка, bundle —
   встраивание (озвучено в I-2).

4. **Q-I4 (объём result-снимка в repro).** Сохранять полный результат всех
   режимов тестера или только активного режима на момент Export? Полный набор =
   тяжелее, но «всё что видел отправитель». *Допущение:* только активный режим
   запроса (минимальный bug-report; кластер G решит про полный golden-снимок).

5. **Q-I5 (verify-tolerance).** `navmesh_hash` — строгий байт-в-байт (любое
   расхождение = MISMATCH), или допускать epsilon на float-полях (другая арка/
   FP)? *Допущение:* строгий байт-в-байт в пределах одной арки/тулчейна (out:
   cross-arch, §2); cross-arch = отдельный warning «другая платформа, repro не
   гарантирован», не fail.

6. **Q-I6 (граница с кластером G).** Где грань: I даёт бинарный verify
   «совпало/нет» + repro-bundle; G строит на этом золотые снимки/регрессии/
   визуальный diff. Подтвердить, что **diff-визуализация и golden-наборы — это G,
   а не I** (I — только repro-контракт + bundle). *Допущение:* да (озвучено в §2 out).

7. **Q-I7 (нужен ли P-API `appendToArchive`?).** Чтобы вложить `repro/` в bundle,
   достаточно ли «положить `repro/` внутрь `.recastscene/` и упаковать целиком»
   (тогда P.`packArchive` ничего не знает про repro — и это чисто кластер I), или
   фундаменту нужен явный hook? *Допущение:* достаточно подкаталога `repro/`
   внутри сцены — **P-API расширять не нужно** (минимизирует связанность). Это
   подтверждает: bundle ≡ packArchive сцены с `repro/`. Согласовать с владельцем P.

## 6. Риски

- **R1 — Блокировка на нереализованном фундаменте.** Sc/P/U ещё не существуют
  (проверено: `persist/`, `scene.zig` отсутствуют). I нельзя начинать раньше
  фундамент-этапов §7.2/§7.4. *Митигация:* строго соблюдать порядок (фундамент
  1→2→4, затем I); проектировать I против *интерфейсов* фундамента (snapshot/
  packArchive/checksum), не против реализации.

- **R2 — Рассинхрон форматов с P.** Если I начнёт писать собственный байт-лейаут
  (volumes/tiles/manifest) вместо P — два источника истины, дрейф версий.
  *Митигация:* I **только оркестрирует** P; единственный *новый* формат I —
  `repro/*` (query/result/meta), и он тоже использует P.checksum/header-конвенцию
  (фундамент 3.b: magic/version/XXH3).

- **R3 — Иллюзия cross-platform детерминизма.** Float-операции (FMA, x87 vs SSE,
  ARM) могут дать иной байт навмеша на другой арке/компиляторе → ложный MISMATCH,
  «у меня не воспроизводится». *Митигация:* §2 явно out cross-arch; `repro.meta`
  пишет арку/тулчейн; mismatch с *другой* аркой = warning, не fail (Q-I5).
  Документировать: repro-гарантия — в пределах одной арки/тега порта.

- **R4 — `.obj` в bundle раздувает файл.** Большие меши встраиваются в каждый
  bundle. *Митигация:* bundle — для bug-report (обычно один кейс, не серия);
  сжатие — если P.`packArchive` сожмёт; иначе YAGNI до подтверждённого размера.

- **R5 — Repro-запрос ссылается на удалённый poly-флаг/area.** Если bundle
  содержит include/exclude-флаги или area, которых нет в реестре получателя
  (но они в `edits/` сцены — значит будут восстановлены). Риск только если флаг
  редактировали после export. *Митигация:* флаги/area едут В bundle (часть
  сцены, `edits/`), запрос ссылается на них по имени/биту — целостно.

- **R6 — Partition-заглушка вводит в заблуждение.** Пользователь меняет
  Partitioning-чекбокс, ждёт другого навмеша, repro «ломается» необъяснимо.
  *Митигация:* Q-I2 — поле informational; verify не винит `partition`, пока
  конвейер его игнорирует; явная подпись в UI/Log.

- **R7 — Раздувание `main.zig` (I-5 arg-parsing).** *Митигация:* парсинг bundle/
  verify — в `repro/cli.zig`, `main.zig` только делегирует (как уже делает с
  `--cam`/`--draw`).

## 7. Этапы реализации (порядок)

Предусловие (фундамент, §7 фундамента): **1** U-shell, **2** Scene-каркас,
**4** Persist per-tile + `write_atomic`/`checksum`/`packArchive`. I стартует
ТОЛЬКО после них.

1. **I-1 Save/Load сцены** (`scene_session.zig`). Оркестрация P-вызовов поверх
   `SceneSnapshot`; команды Save/Load через U-shell; сброс инструментов + (re)build
   при Load. *Разблокирует:* всё остальное в I; F (save после правки), G (снимки).
   *Критерий:* save→перезапуск→load даёт идентичную сцену (визуально + хэши edits).

2. **I-3 repro-манифест + navmesh-hash** (`repro_manifest.zig`). `repro.meta`
   (версии/settings/geom_hash/navmesh_hash через P.checksum); детерминированный
   `navmeshHash` (конкатенация tile-blob по `(tx,ty,layer)`). Без UI-verify пока —
   только запись/чтение. *Разблокирует:* verify в I-2/I-5. *Критерий:* два Build
   подряд из одной сцены → одинаковый `navmesh_hash` (доказательство детерминизма).

3. **I-4 repro-запрос** (`query_record.zig`). Снять/восстановить состояние
   `tool_navmesh_tester`; reuse `testcase.zig` pf/rc для txt-совместимости;
   бинарь для прочих режимов. *Критерий:* export→import восстанавливает точки/
   флаги/режим, пересчёт даёт тот же результат.

4. **I-2 bundle** (`bundle.zig`). `exportBundle` = собрать сцену (I-1) + `repro/`
   (I-3+I-4) → `packArchive`; `importBundle` = `unpackArchive` → load (I-1) →
   restore query (I-4) → verify (I-3) с итогом в Log. *Критерий:* bundle с одной
   машины разворачивается на другой (та же арка) с `repro OK`.

5. **I-5 CLI** (`repro/cli.zig`). Позиционный `<bundle>`, `--verify=` (headless
   exit-code), `--export-bundle=`; делегирование из `main.zig`. *Разблокирует:*
   CI-регрессии (G), headless (H). *Критерий:* `recast_demo --verify=x.recastbundle`
   возвращает 0/≠0 без окна.

6. **(опц.) Полировка**: txt-экспорт repro-запроса в `TestCases/`, диалоги путей,
   drag-onto-window. YAGNI до запроса владельца.

Критический путь: **фундамент(1,2,4) → I-1 → I-3 → I-4 → I-2 → I-5**.
