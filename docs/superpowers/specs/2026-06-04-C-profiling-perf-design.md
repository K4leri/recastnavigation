## 1. Цель и ценность для developer-user

Сейчас перф-сигнал в демо разрознен и эфемерен:

- **Build-time** — одна строка `Build Time: {d:.1}ms` (`sample_solo.zig:556`,
  поле `build_time_ms`, `sample_solo.zig:49`), посчитанная одним таймером вокруг
  всего `buildNavMesh` (`timer.readMs()`, `sample_solo.zig:198`). Разбивки **по
  стадиям нет**: `Context.startTimer`/`stopTimer` — это TODO-заглушки
  (`src/context.zig:94-99`), а `TimerLabel` (28 меток, `src/context.zig:11-40`)
  определён, но **нигде не наполняется** (вызовы закомментированы, напр.
  `src/recast/compact.zig:59`). История прогонов не хранится — каждый Build
  затирает прошлое число.
- **Render-перф** — счётчики `dd_gl.draw_calls`/`verts_uploaded`
  (`debug_draw_gl.zig:104-105`, инкремент `:339-340`) сбрасываются раз в кадр
  (`main.zig:336-337`) и видны только через `--bench`-режим (`main.zig:1026-1039`,
  печать в stderr) или Tracy-plot (`main.zig:629-630`). В GUI их нет.
- **Tracy** — есть тонкая обёртка (`tracy.zig`), но это внешний профайлер,
  требующий отдельного клиента и сборки `-Dtracy`; для быстрой in-app оценки
  «сколько стоит эта сцена» не подходит.
- **Запросы (pathfind)** — `tool_navmesh_tester.zig` гоняет одиночные запросы;
  агрегированной статистики (p50/p95/p99, узлов на запрос) **нет**.
- **Память** — размеры тайлов/навмеша/полимеша считаются при сборке
  (`sample_tile.zig:208-216` — `vert_count`/`poly_count`/`detail_*`), но нигде не
  агрегируются и не показываются как бюджет.

**Ценность.** Кластер C превращает «магическое число Build Time» в инструмент
инженерного анализа: *где* в конвейере уходит время, *сколько* стоит запрос на
этом навмеше под нагрузкой, *сколько памяти* съедают тайлы/tilecache, и *какой
партиционер* (watershed / monotone / layers) даёт лучший trade-off скорость↔
качество для конкретной сцены. Это база для оптимизации настроек сборки и для
регрессионного контроля перфа (стыкуется с кластером G).

**Важное ограничение на старте (см. §5 Q1).** Полноценная «разбивка по стадиям»
требует, чтобы ядро `src/*` наполняло `Context`-таймеры. Сейчас этого нет.
Кластер C **не переписывает конвейер ядра**; вместо этого на этапе 1 он измеряет
стадии на **demo-уровне** — оборачивая уже-раздельные вызовы стадий в сэмпле
(`sample_solo.zig:255-297` — rasterize / compact / erode / partition / contours /
polymesh / detail идут отдельными вызовами, их легко обернуть `nanoTime`). Это
faithful-safe и даёт 8-10 крупных стадий без правок ядра. Тонкая
суб-стадийная разбивка (28 `TimerLabel`) — опциональный этап, см. Q1/§7.

## 2. Scope (in / out / YAGNI)

### Входит (in)

- **Demo-уровневый профайлер сборки**: обёртка раздельных стадий
  `buildNavMesh`/`buildTileMesh` в `nanoTime`-таймеры, агрегатор результатов,
  история N последних прогонов (ring buffer), панель с таблицей+баром стадий.
- **Бенчмарк запросов**: прогон N случайных (или сетки) pathfind-запросов на
  текущем навмеше, сбор латентностей и счётчика узлов/запрос, перцентили
  p50/p95/p99, гистограмма. Переиспользует существующий `dt.NavMeshQuery` и
  фильтр из tester'а.
- **Бюджет памяти**: агрегатор размеров (per-tile data_size, суммарный navmesh,
  tilecache compressed vs raw, полимеш/детейл) — таблица + текущий пик.
- **Сравнение партиционеров**: прогон одной сцены через watershed / monotone /
  layers, сводка время-сборки × качество (число полигонов, число регионов,
  фрагментация) — таблица сравнения.
- **Tile-cost heatmap**: 2D/3D-раскраска тайлов по времени построения и/или
  размеру (через R.color_scheme + R.legend), для Tile/Temp-сэмплов.
- **Render-перф HUD**: вынос `draw_calls`/`verts_uploaded`/frame-time в overlay
  (через R.overlay) и в перф-панель (через U).
- Регистрация перф-инструмента/панели через **U-shell** (`tool_registry`/`panel`).
- Опциональная связка с **Tracy** (плоты бенчмарков), когда `-Dtracy=true`.

### Не входит (out)

- Изменение faithful-ядра `src/*` ради таймеров (наполнение `TimerLabel` в ядре —
  **только** если владелец одобрит Q1; по умолчанию demo-уровень).
- Замена/расширение `bench/*` (это headless micro-bench для CI, отдельный контур;
  кластер C — *in-app* профилирование, не CI-бенч).
- Полноценный встроенный профайлер-флеймграф/семплер CPU (это Tracy; не дублируем).
- Персистентность результатов профиля на диск как часть фундамента (P ○ —
  опционально; экспорт CSV/JSON — YAGNI-минимум, см. Q4).
- Сравнение Zig-порта против C++ reference (это `recast-benchmarking`-skill /
  Tracy-сценарии, вне демо).
- GPU-таймеры (`glQueryCounter`/timer queries) — YAGNI; frame-time по CPU-стенке.

### YAGNI (явно отложено)

- Авто-tuning настроек сборки по результату профиля.
- Регрессионные пороги/алерты перфа (это кластер G на готовых данных C).
- Мульти-thread tile-build профилирование (сборка сейчас однопоточная).
- Дифф двух сохранённых профилей (нужна P; отложить до спроса).

## 3. Фичи (приоритезированы)

Приоритет: **P0** — ядро ценности, делаем первым; **P1** — высокая ценность;
**P2** — приятно иметь.

### C1 (P0) — Профайлер стадий сборки + история прогонов

- **Что.** Замер времени каждой крупной стадии конвейера (rasterize, filter,
  compact, erode, mark-volumes, partition, contours, polymesh, detail,
  detour-data) при каждом Build; агрегат в структуру `BuildProfile`; кольцевой
  буфер последних N прогонов (напр. N=16). Панель: таблица «стадия | мс | %» +
  горизонтальный stacked-bar, и спарклайн total по истории.
- **Зачем.** Заменить единственное число `build_time_ms` на разбивку — видно,
  что доминирует (обычно regions/contours/detail), и как меняется total при
  правке настроек/партиционера. Основа для C4 и C5.
- **UX.** Новая секция/панель «Profiler» (через U-shell `panel`). После Build —
  таблица заполняется автоматически. Столбики стадий цветные (по `ColorScheme`-
  палитре R, но это 2D-бар, не навмеш). История — выпадающий список прогонов +
  спарклайн; клик по прогону показывает его разбивку. Кнопка «Clear history».
- **Данные.**
  - Источник: обёртка раздельных вызовов стадий в `sample_solo.buildNavMesh`
    (`sample_solo.zig:255-297`) и `sample_tile.buildTileMesh`
    (`sample_tile.zig:117-232`) в `nanoTime` (паттерн `impl.nanoTime()`,
    `main.zig:1008`; `timer.readMs()` уже используется, `sample_solo.zig:198`).
  - `BuildProfile { stages: [N]StageTime, total_ms, gen, partition: SamplePartitionType, n_polys, n_regions, n_tiles, mem: MemBudget }`.
  - Для Tile-сборки стадии суммируются по тайлам + хранится per-tile разбивка для
    C5 (heatmap).
- **Зависимости.**
  - Ядро: читает `pmesh.polyCount()`/`vertCount()` (`sample_tile.zig:197,208`),
    `SamplePartitionType` (`sample.zig:14-17`), `build_time_ms`/`build_gen`
    (`sample_solo.zig:49,199`). **Без правок ядра** (стадии уже раздельны).
  - Фундамент: **U** (`panel.zig` для окна Profiler, `tool_registry` не нужен —
    это панель, не инструмент); **R** опц. (палитра для бара). Sc/P — нет.
  - *Не-реализованный фундамент:* U-shell `panel.zig` (фундамент §7 шаг 1) —
    **блокер для аккуратной интеграции**; до него — временно инлайн `floatingWindow`
    по образцу `main.zig:643-659`.

### C2 (P0) — Бенчмарк запросов (pathfind) с перцентилями

- **Что.** Кнопка «Run query bench»: генерирует K случайных пар (start,end) на
  навмеше (через `findRandomPoint`/равномерно по тайлам), прогоняет `findPath`
  (+опц. `findStraightPath`), собирает на каждый запрос: латентность (нс), число
  посещённых узлов, длину пути / число полигонов, success/fail. Считает
  p50/p95/p99/min/max/avg, success-rate, среднее узлов. Гистограмма латентности.
- **Зачем.** «Сколько стоит запрос на этом навмеше» — ключевая метрика для
  игрового бюджета. Перцентили важнее среднего (хвосты = лаги).
- **UX.** Панель «Query Bench» (через U): слайдеры K (число запросов), seed,
  чекбоксы (straight path on/off, использовать include/exclude-флаги тестера).
  «Run» → прогресс → таблица перцентилей + гистограмма (2D-бары) + опц.
  визуализация worst-case пути на сцене (через R.highlight — подсветка полигонов
  самого долгого запроса).
- **Данные.**
  - Источник: `dt.NavMeshQuery` (как в `tool_navmesh_tester.zig`), фильтр
    `QueryFilter` (include/exclude из реестра флагов). Счётчик узлов — из node
    pool запроса (проверить, экспонирован ли; если нет — см. Q3).
  - `QueryBenchResult { latencies_ns: []u64, nodes: []u32, ok: []bool, percentiles }`.
- **Зависимости.**
  - Ядро: `src/detour` (`NavMeshQuery.findPath`/`findRandomPoint`/
    `findStraightPath`, `QueryFilter`). **Узлы/запрос** могут потребовать
    геттера на pool (Q3) — *возможная* минимальная правка ядра (геттер, не логика).
  - Фундамент: **U** (панель), **R.highlight** (worst-case путь, §3.c). Sc — ○
    (фильтр-флаги из реестра, но напрямую). P — нет.

### C3 (P1) — Бюджет памяти

- **Что.** Агрегатор размеров: per-tile `data_size` (из `addTile`-данных,
  `sample_tile.zig:232`), суммарный navmesh, полимеш (`vert_count`/`poly_count`/
  `detail_*`, `sample_tile.zig:208-216`), tilecache compressed vs raw (Temp-сэмпл,
  компрессор сейчас no-op — raw==compressed, честно показать), текущий пик.
  Таблица + (опц.) bar по тайлам.
- **Зачем.** Память тайлов/tilecache — лимитирующий ресурс на больших мирах;
  compressed-vs-raw показывает выигрыш компрессии (или его отсутствие — сейчас
  no-op, FEATURES §2 Temp).
- **UX.** Секция «Memory» в перф-панели: список (Navmesh total, PolyMesh,
  DetailMesh, Tiles N × avg, TileCache raw/compressed/ratio, Peak).
- **Данные.** Источник — поля сборки сэмплов; tilecache — `src/detour_tilecache`
  (размеры слоёв). `MemBudget` встраивается в `BuildProfile` (C1).
- **Зависимости.** Ядро: чтение размеров (без правок). Фундамент: **U**.
  P — ○ (если захотим сохранять бюджет — отложено, Q4).

### C4 (P1) — Сравнение партиционеров (скорость × качество)

- **Что.** Кнопка «Compare partitioners»: пересобирает текущую сцену тремя
  способами (watershed/monotone/layers, `sample.zig:14-17`), для каждого
  фиксирует: build-time (из C1), число полигонов, число регионов, метрику
  фрагментации (напр. avg polys/region, доля мелких регионов, число контуров).
  Таблица сравнения 3×N-метрик с подсветкой лучшего.
- **Зачем.** Watershed качественнее, monotone быстрее, layers для тайлов — но
  trade-off зависит от сцены. Дать developer-user объективную сводку вместо
  «на глаз».
- **UX.** Кнопка в перф-панели → прогон трёх сборок (с прогрессом, т.к.
  пересборка медленная) → таблица. Опц. кнопка «Apply best» (ставит
  `s.partition_type`). Текущий тип помечен.
- **Данные.** Прогоняет `buildNavMesh` трижды со сменой `s.partition_type`
  (`sample_solo.zig:268-278` уже ветвится). Метрики качества — из `pmesh`/`cset`
  (`npolys`, число контуров) + region-count (из chf после partition).
- **Зависимости.** Ядро: чтение `pmesh`/`cset`/region-count (region-count может
  требовать геттера — Q3). Фундамент: **U**. **R** опц. (раскрасить навмеш по
  region для визуального сравнения фрагментации — `ColorScheme.region`, §3.c).
  *Поведение:* по завершении восстановить исходный `partition_type` и навмеш
  (или оставить best, по UX-выбору).

### C5 (P2) — Tile-cost heatmap

- **Что.** Для Tile/Temp-сэмплов: раскраска каждого тайла по времени его
  построения и/или итоговому размеру (per-tile данные из C1). Шкала
  cold→hot. 2D-миникарта тайлов и/или 3D-оверлей поверх навмеша.
- **Зачем.** Найти «дорогие» тайлы (плотная геометрия, много регионов) — точки
  оптимизации tile-size / геометрии.
- **UX.** Переключатель режима heatmap (time / size) в перф-панели; раскраска
  тайлов на сцене через R.color_scheme (новая схема `cost`/`height`-подобная) +
  R.legend (шкала). 2D-миникарта — опц.
- **Данные.** Per-tile `{tx, ty, build_ns, data_size, n_polys}` из C1
  (Tile-сборка цикл `sample_tile.zig:104` по tx/ty).
- **Зависимости.** Ядро: per-tile метрики (из C1, без правок). Фундамент:
  **R.color_scheme** + **R.legend** + **R.overlay** (§3.c) — heatmap-схема
  применяется per-tile при обходе. **U** (переключатель). *Сильно опирается на
  ещё-не-реализованный R (фундамент §7 шаг 3) — блокер для C5.*

### C6 (P2) — Render-перф HUD

- **Что.** Вынести в GUI то, что сейчас только в `--bench`/Tracy: frame-time
  (мс/FPS), `draw_calls`, `verts_uploaded` — постоянный overlay-уголок + строка
  в перф-панели. Опц. мини-график frame-time (как crowd perf-graph,
  FEATURES §4.5 «Show Perf Graph»).
- **Зачем.** Видеть стоимость отрисовки текущего debug-режима (воксели = сотни
  тысяч боксов, см. `debug_draw_gl.zig:93-94`) прямо в окне.
- **UX.** Чекбокс «Show perf HUD» (секция Show, как `app_state.zig:15-18`);
  overlay через R.overlay (worldspace/screen-текст, `ui.screenText`).
- **Данные.** `dd_gl.draw_calls`/`verts_uploaded` (`debug_draw_gl.zig:104-105`),
  frame-time из `impl.nanoTime()` дельты (`main.zig:1008,1027`).
- **Зависимости.** Фундамент: **R.overlay** (§3.c), **U** (чекбокс). Ядро — нет.

**Порядок ценности:** C1 → C2 → C3 → C4 → C5 → C6. C1/C2 — самостоятельная
ценность без R; C5 жёстко требует R; C6 — косметика на готовом R.

## 4. Архитектура

Всё — **demo-уровень** (`demo/src/*`), faithful-ядро `src/*` не трогаем (искл.
возможные read-only геттеры по Q3). Конвенции `.agent/project_structure.md`:
snake_case, usize-геттеры на demo-структурах, faithful-касты на границе.

### Новые файлы

```
demo/src/profiling/
├── build_profile.zig    # BuildProfile + StageTime + MemBudget; ring-buffer истории
│                         #   (последние N прогонов). Чистые данные, без UI.
├── stage_timer.zig      # тонкий scoped-таймер на impl.nanoTime() для обёртки
│                         #   стадий в сэмплах (НЕ путать с src/context.zig ScopedTimer-
│                         #   заглушкой; этот реально меряет). C1.
├── query_bench.zig      # генерация K запросов, прогон findPath, сбор латентностей/
│                         #   узлов, перцентили p50/p95/p99, гистограмма. C2.
├── partition_compare.zig# прогон 3 партиционеров, сбор time×quality. C4.
├── mem_budget.zig       # агрегатор размеров tile/navmesh/polymesh/tilecache. C3.
└── perf_panel.zig       # UI: панель Profiler (таблицы/бары/гистограммы/heatmap-
                          #   контролы) поверх dvui; регистрируется через U-shell panel.
```

Группировка в `profiling/` — по образцу фундаментских `persist/`, `render/`,
`shell/` (фундамент §3.b/§3.c/§3.d). Данные (`build_profile`/`query_bench`/
`mem_budget`/`partition_compare`) отделены от UI (`perf_panel`) — headless-friendly
(можно дёргать из `--bench`/тестов без GUI).

### Точки интеграции

| Регистрация панели | `main.zig:643-659` (floatingWindow), `app_state.zig:15-18` (show-флаги) | через U-shell `panel.zig` (когда готов); до него — инлайн |

### Опора на подсистемы фундамента (НЕ передизайниваем их)

- **U-shell (§3.d):** перф-панель регистрируется как `PanelDesc` (`panel.zig`,
  фундамент §3.d) — `{ title:"Profiler", rect_fn, visible:*bool, draw_fn }`. Ввод
  гейтится через `input_gate` (`ui_mouse`/`ui_keyboard`, фундамент §3.d). Перф —
  это **панель**, не инструмент, поэтому `tool_registry` не задействуется (искл.:
  если query-bench-worst-case-визуализация станет интерактивным инструментом —
  тогда `ToolEntry`).
- **R (§3.c):** C5 heatmap использует `color_scheme.zig` (новая схема стоимости —
  фундамент уже закладывает `cost`/`height` в enum `ColorScheme`, §3.c) +
  `legend.zig` (шкала) + `overlay.zig` (аннотации тайлов). C2 worst-case — через
  `highlight.zig`. C6 HUD — через `overlay.zig`. **Сами механизмы R — фундамент,
  кластер C только потребитель** (граница Q7 фундамента).
- **Sc/P:** не используются (Sc ○ — фильтр-флаги читаем напрямую из реестра;
  P ○ — экспорт профиля отложен, Q4).

### Tracy-связка (опционально)

Когда `-Dtracy=true`: `query_bench`/`partition_compare` дополнительно эмитят
`tracy.plotF` (обёртка `tracy.zig:39`) и `tracy.zone` вокруг стадий — для внешнего
анализа. Когда `-Dtracy=false` — zero-cost (stub, `tracy.zig:7`). Это **дополнение**,
не зависимость: вся ценность C доступна без Tracy.

## 6. Риски

- **C-R1 — Блокировка на ещё-не-реализованном фундаменте.** C5 жёстко требует R
  (`color_scheme`/`legend`/`overlay`, фундамент §7 шаг 3); C1/C6 удобнее с U
  (`panel`, §7 шаг 1). *Митигация:* C1/C2/C3/C4 не зависят от R и дают ценность
  раньше; до U-`panel` — инлайн `floatingWindow` (`main.zig:643-659`), позже
  мигрировать. C5 ставим после R.
- **C-R2 — Соблазн править ядро ради таймеров.** Тонкая разбивка (Q1) тянет
  правки в `src/*` (28 call-site). Риск регрессий faithful-поведения. *Митигация:*
  стартуем с demo-стадий; наполнение ядра — только по явному Q1-одобрению, под
  регрессионными тестами (`test/integration/`), таймеры за флагом
  `timer_enabled` (`context.zig:53`, уже есть).
- **C-R3 — Шум измерений.** Один build-прогон зашумлён (кэш, планировщик,
  idle-cap кадров `main.zig:1006-1014`). *Митигация:* история N прогонов + показ
  min/median, не одно число; для C2 — warmup-итерации (как `bench/recast_bench.zig:10`
  `warmup_iterations`). Документировать, что это wall-clock, не изолированный
  micro-bench.
- **C-R4 — Стоимость C2/C4 (пересборки/тысячи запросов) фризит UI.** Сборка
  медленная; C4 = ×3 сборки. *Митигация:* прогон с прогресс-индикатором; гонять
  пачками между кадрами или принять кратковременный фриз с предупреждением;
  K-запросов C2 ограничить разумным дефолтом. (Async/threaded — YAGNI, сборка
  однопоточная.)
- **C-R5 — Узлы/регионы недоступны без правки ядра (Q3).** *Митигация:* см. Q3 —
  минимальные read-only геттеры либо demo-уровневый подсчёт; деградировать
  gracefully (скрыть метрику, если недоступна).
- **C-R6 — Per-tile тайминги искажены border/overlap.** Тайлы строятся с
  расширением границ (`sample_tile.zig` `border_size`), геометрия перекрывается —
  «время тайла» включает общую работу. *Митигация:* документировать как
  «relative cost», не абсолютную атрибуцию; для heatmap (C5) этого достаточно.
- **C-R7 — Tilecache compressed==raw (no-op компрессор).** C3 покажет ratio=1.0
  (FEATURES §2 Temp: компрессор без сжатия). *Митигация:* честно показать «no
  compression», не выдавать за выигрыш.

## 7. Этапы реализации (порядок)

Порядок: ранняя ценность без блокеров фундамента → затем фичи на R.

1. **C1-core — данные профиля сборки (demo-стадии).** `build_profile.zig` +
   `stage_timer.zig`; обернуть стадии в `sample_solo.zig:255-297` и
   `sample_tile.zig:117-232`; ring-buffer истории. *Без UI, без R, без ядра.*
   Проверяемо через `--bench`/тест. **Разблокирует:** C4, C5 (данные), C6.
2. **C1-ui — панель Profiler.** `perf_panel.zig`: таблица стадий + stacked-bar +
   история. Через U-`panel` если готов, иначе инлайн `floatingWindow`. **+C3**
   (`mem_budget.zig` — секция Memory, дешёвая, данные уже при сборке).
3. **C2 — query bench.** `query_bench.zig` (генерация+прогон+перцентили+
   гистограмма) + секция в `perf_panel`. Узлы/запрос — по Q3. Worst-case
   подсветка — опц., через R.highlight (если R готов).
4. **C4 — partition compare.** `partition_compare.zig` (×3 сборки, time×quality)
   + таблица сравнения. Опирается на C1-core (тайминги) и метрики качества (Q7).
5. **C5 — tile-cost heatmap.** *Требует R* (фундамент §7 шаг 3). Новая heatmap-
   схема в `color_scheme` (cost), `legend`-шкала, per-tile раскраска. **Блокер:**
   готовность R.
6. **C6 — render-перф HUD.** *Требует R.overlay.* frame-time/draw_calls/verts в
   overlay+панель. Косметика на готовом R.
7. **(опц., по Q1) Тонкая разбивка стадий в ядре.** Реализовать `startTimer`/
   `stopTimer` (`context.zig:94-99`), раскомментировать call-site (28 `TimerLabel`).
   **Только по одобрению Q1**, под регрессионными тестами. Расширяет C1 без
   изменения его UI.

**Критический путь ценности:** 1 → 2 → 3. Шаги 5-6 ждут R; шаг 7 — опционален и
требует одобрения правки ядра.
