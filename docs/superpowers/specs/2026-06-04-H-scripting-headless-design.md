## 1. Цель и ценность для developer-user

Сейчас демо (`demo/src/`) — **только GUI**: единственный исполняемый артефакт
`recast_demo` собирается из `demo/src/main.zig` и на старте безусловно поднимает
glfw-окно + OpenGL-контекст (`main.zig:56-115`), линкует `dvui`/`glfw`/`zgl`
(`build.zig:303-329`). Любой запрос к навмешу (`findPath`/`findStraightPath`/
`findNearestPoly`) живёт внутри интерактивного инструмента `tool_navmesh_tester`
(`tool_navmesh_tester.zig:120-201`), запускается мышью и осмысленен только на
экране. Параметры сборки (`CommonSettings`, `sample.zig:61-80`) меняются
слайдерами; результат — глазами. Для CI/регрессий есть лишь графические флаги
`--bench`/`--cyclemodes`/`--draw=`/`--cam=` (`main.zig:241-266`), которые всё
равно требуют GL-контекста.

**Кластер H** превращает движок демо в **headless-инструмент**: тот же
build-конвейер и те же Detour-запросы — но без окна, управляемые конфигом и
скриптом, с машиночитаемым выводом (JSON/CSV/exit-code). Это даёт
developer-user три встроенных сценария, которых сейчас нет:

- **Воспроизводимый прогон из терминала / CI.** «Загрузи сцену X, собери с
  параметрами Y, ответь — находится ли путь A→B и сколько он стоит» — одной
  командой, без человека у экрана. Exit-code и JSON вместо скриншота.
- **Батч-матрица параметров.** «Перебери `cell_size` ∈ {0.2,0.3,0.5} ×
  `agent_radius` ∈ {0.4,0.6} → таблица (poly-count, build-ms, успех набора
  запросов)». Сейчас это ручной перебор слайдеров.
- **Скрипт/сценарий запросов.** JSON-сценарий «N стартов/концов + фильтры
  include/exclude → пути/raycast/distance-to-wall» прогоняется пакетом и выдаёт
  метрики. Обобщение того, что `testcase.zig` уже делает для `TestCases/*.txt`,
  но управляемое и расширяемое.

Критическая предпосылка (из фундамента §4, строка H и §3.a): **`area_types` и
`poly_flags` сейчас module-global** (`area_types.zig:50-51` `var types`/
`initialized`; `poly_flags.zig:40-41` `var flags`/`initialized`; доступ через
`ensureInit()`). Headless-инстансы (две сцены в одном процессе, батч-воркеры)
**конфликтуют по этому глобальному состоянию** — поэтому H **фундаментально
зависит** от выноса реестров в инстанс `Scene` (фундамент 3.a, этап 5). До этого
выноса headless работает только в режиме «одна сцена на процесс» (см. §5 Q-H1).

Ценность: navmesh из «интерактивной демки» становится **тестируемым,
скриптуемым, CI-пригодным** артефактом — регрессии параметров сборки и
корректности запросов ловятся автоматически, а не «на глаз».

## 2. Scope (in / out / YAGNI)

### Входит (in)

- **Headless entry-point**: отдельная точка входа `demo/src/headless.zig`,
  которая НЕ инициализирует glfw/GL/dvui, поднимает только ядро
  (`Scene` + сэмпл-сборку + Detour-запросы) и пишет в stdout/файлы. Подключается
  как отдельный build-step `headless` / отдельный exe (см. §4) — GUI-exe
  `recast_demo` НЕ меняется по поведению.
- **Конфиг-файл сцены/прогона** (синергия с Persist 3.b): декларативное описание
  «что грузить, как собрать, что запросить, куда писать». Формат — JSON
  (Zig 0.16 `std.json`), маленький стабильный набор полей.
- **Загрузка сцены headless** через Persist (3.b): `.recastscene/`-контейнер
  (геометрия `.gset`+`.obj` + edits: areas/flags/volumes/offmesh). Деградация:
  пока 3.b не готов — fallback на текущий `InputGeom.loadMesh(.obj)`
  (`input_geom.zig:77`) + параметры из конфига (см. §3.1 «зависимость»).
- **Батч-матрица**: декартова матрица перечислимых параметров сборки →
  таблица метрик (CSV + JSON): на каждую ячейку — статус сборки, poly/vert-count,
  build-ms, и агрегат набора запросов (сколько путей найдено, суммарная стоимость).
- **Скрипт/сценарий запросов** (JSON, не REPL — см. YAGNI): массив запросов
  `{type, start, end, include, exclude, ...}` → массив результатов
  (`found`, `path_len`, `straight_points`, `cost`, `polys`, `nearest_ref`, ...).
  Типы запросов — подмножество уже реализованных в tester
  (`tool_navmesh_tester.zig`): `findPath`, `findStraightPath`, `raycast`,
  `findDistanceToWall`, `findNearestPoly`.
- **Машиночитаемый вывод**: JSON-отчёт прогона + CSV-таблица батча + осмысленный
  process exit-code (0 = все запросы прошли ожидание / сборка успешна; ≠0 —
  иначе), чтобы CI мог `assert`.
- **Опциональные офф-скрин скриншоты** — ТОЛЬКО как явный отдельный режим, если
  владелец подтвердит (см. §5 Q-H4); по умолчанию **out** (требует GL).

### Не входит (out / YAGNI)

- **Встроенный язык/REPL/интерактивная оболочка.** Идея «мини-скрипт/REPL»
  из задания **сознательно сведена к декларативному JSON-сценарию** (YAGNI):
  парсер выражений, переменные, циклы внутри скрипта — это отдельный язык,
  несоразмерный задаче. Матрицу даёт батч-конфиг, последовательность — массив
  запросов. REPL (stdin-команды построчно) — отдельный поздний вопрос (§5 Q-H5),
  не в стартовом scope.
- **Параллельные воркеры/threadpool батча.** Стартуем последовательно (один
  процесс, цикл по ячейкам). Параллелизм требует завершённого выноса
  module-global (3.a этап 5) И thread-safe реестров — отдельный этап (§7 этап 4),
  не в MVP.
- **Рендер/overlay (3.c) и UI-shell (3.d).** Кластер H их НЕ использует
  (фундамент §4: H по R/U = `○`). Никаких цветовых схем/легенд/панелей.
- **Новые форматы импорта/экспорта** (это кластер D). H читает то, что умеет
  Persist (3.b) и текущий `InputGeom`; не вводит glTF/новые форматы.
- **Изменение формата `.gset`/MSET** — расширяемся только sidecar-ами Persist,
  как требует фундамент (риск R7 фундамента).
- **Изменение faithful-ядра `src/*`.** Запросы идут через существующий публичный
  API `recast.detour` (`NavMeshQuery`/`QueryFilter`) — ядро не трогаем.
- **Демоны/watch-режим/сервер.** Один прогон = один процесс. Долгоживущий
  сервис — вне scope.
- **Скриншоты headless по умолчанию** (нужен GL — противоречит «headless»).

## 3. Фичи (приоритезированы)

Приоритет: **P0** — ядро кластера (без него H не существует); **P1** — даёт
основную ценность батча/скрипта; **P2** — удобство/полировка.

### 3.1 [P0] Headless entry-point + загрузка сцены без GL

- **Что.** Новая точка входа `demo/src/headless.zig` со своей `main`, которая
  поднимает `Scene` (фундамент 3.a) → строит навмеш существующим сэмпл-конвейером
  → завершается, **не создавая** glfw-окна/GL-контекста/dvui. Отдельный build-step
  `headless` и отдельный exe (`recast_headless`).
- **Зачем.** Сейчас единственная `main` (`main.zig:56`) безусловно вызывает
  `zglfw.init()` / `Window.create` / `zgl.loadExtensions` (`main.zig:74-106`) —
  на машине без дисплея/GL это просто не запустится. Без отдельного входа
  «headless» невозможен в принципе.
- **UX.** `recast_headless --config run.json` (или подкоманды, §4). Никакого
  окна; весь вывод — stdout/файлы + exit-code.
- **Данные.** Вход: конфиг (§3.2). Внутри: `Scene` (geom + areas + flags +
  `CommonSettings`), затем сэмпл (`SampleSolo`/`SampleTile`/`SampleTempObstacles`)
  → `dt.NavMesh`. Выход: JSON-отчёт сборки (poly/vert-count, build-ms, успех).
- **Зависимости.**
  - **Ядро `src/*`:** `recast.recast` (build-конвейер), `recast.detour`
    (`NavMesh`). Через сэмплы — без прямого касания ядра.
  - **Фундамент Scene (3.a):** `Scene` как контейнер geom/areas/flags/settings.
    **Блокер:** сэмплы сейчас конструируются с `*DebugDrawGL`
    (`sample_solo.zig:46,61`; `SampleSolo.init(alloc, bctx, dd_gl)`), хотя
    `build()`/`doBuild()` НЕ используют `dd_gl` (он трогается только в
    `render()`, `sample_solo.zig:365`). Требуется **либо** допустить `null`/
    stub-`DebugDrawGL` в headless, **либо** расщепить конструктор сэмпла на
    build-часть (без GL) и render-часть. См. §4 «Точка интеграции: сэмплы без GL»
    и §5 Q-H2.
  - **Фундамент Persist (3.b):** загрузка `.recastscene/` (geom+edits). **Не
    готов на старте H** → fallback: `InputGeom.loadMesh(.obj)` (`input_geom.zig:77`)
    + области/флаги по умолчанию (`ensureInit`-сиды) + `CommonSettings` из конфига.
    Полная загрузка сцены — после 3.b (этап 4 фундамента).
  - **`io_util.zig`** (`readWholeFile`, `PerfTimer`) для чтения конфига и замера
    build-ms (тот же `PerfTimer.start()`, что в `sample_solo.zig:150`).

### 3.2 [P0] Конфиг-файл прогона (JSON)

- **Что.** Декларативный JSON, описывающий один headless-прогон: что грузить,
  как собрать, какие запросы выполнить, куда писать. Парсится `std.json`
  (Zig 0.16). Маленький фиксированный набор полей, версионируется (`"version": 1`).
- **Зачем.** Воспроизводимость и CI: прогон описан файлом в репозитории, а не
  цепочкой CLI-флагов. Синергия с Persist: конфиг ссылается на `.recastscene/`
  по пути.
- **UX / форма (дизайн, не финальная схема).**
  ```jsonc
  {
    "version": 1,
    "scene": "assets/dungeon.recastscene",   // путь к контейнеру (3.b); или ниже:
    "geom":  "assets/dungeon.obj",            // fallback пока 3.b нет (3.1)
    "sample": "solo",                          // solo | tile | temp
    "settings": { "cell_size": 0.3, "agent_radius": 0.6, "verts_per_poly": 6 },
    "queries": [ /* §3.4 */ ],
    "batch":   { /* §3.3, опц. — взаимоисключающе с одиночным settings */ },
    "out": { "report": "out/report.json", "table": "out/batch.csv" }
  }
  ```
- **Данные.** `settings` маппится 1-в-1 в `CommonSettings` (`sample.zig:61-80`),
  отсутствующие поля → дефолты структуры (faithful). `sample` выбирает
  реализацию. Неизвестные поля — ошибка с понятным сообщением (строгий парс) или
  warning (см. §5 Q-H3).
- **Зависимости.** Только `std.json` + `io_util`. Опирается на Scene-API (3.a)
  для применения `settings`/areas/flags. Persist (3.b) — для `scene`-пути.

### 3.3 [P1] Батч-матрица параметров → таблица метрик

- **Что.** Блок `batch` в конфиге задаёт перечни значений для подмножества полей
  `CommonSettings`; headless строит **декартово произведение**, на каждой ячейке
  пересобирает навмеш, прогоняет общий `queries`-набор и пишет строку таблицы.
- **Зачем.** Систематический подбор/регрессия параметров сборки: «как
  `cell_size`/`region_min_size` влияют на poly-count и на разрешимость набора
  запросов». Сейчас — только ручной перебор слайдеров.
- **UX / форма.**
  ```jsonc
  "batch": {
    "axes": {
      "cell_size":   [0.2, 0.3, 0.5],
      "agent_radius":[0.4, 0.6]
    }
    // 3 × 2 = 6 ячеек; базовые значения берутся из "settings", оси переопределяют
  }
  ```
  Вывод — CSV (одна строка на ячейку) + параллельный JSON-массив:
  ```
  cell_size,agent_radius,build_ok,polys,verts,build_ms,queries_ok,queries_total,total_cost
  0.2,0.4,1,1873,3201,142.3,8,8,57.2
  ...
  ```
- **Данные.** Оси — только числовые/перечислимые поля `CommonSettings`
  (`cell_size`/`cell_height`/`agent_*`/`region_*`/`edge_*`/`verts_per_poly`/
  `detail_*`); `partition_type` — как строковая ось (`watershed|monotone|layers`,
  `sample.zig:14-18`). Метрики ячейки: `build_ok`, `pmesh.polyCount()`/
  `vertCount()` (usize-геттеры по конвенции CLAUDE.md), `build_time_ms`
  (`sample_solo.zig:198`), агрегат запросов (§3.4).
- **Зависимости.** Scene (3.a) — пересоздание/переприменение `settings` на каждой
  ячейке (важно: чистый ресет промежуточных результатов — `cleanup()`,
  `sample_solo.zig:69`). **Module-global реестры (3.a-блокер):** при батче в одном
  процессе area/flags ДОЛЖНЫ быть инстансовыми, иначе ячейки делят глобальное
  состояние (в MVP последовательного батча это терпимо — реестр один и тот же; но
  для будущего параллелизма обязателен вынос). Persist (3.b) — опционально, если
  ячейка сохраняет золотой снимок (кластер G).

### 3.4 [P1] Сценарий запросов (JSON) + агрегаты

- **Что.** Массив `queries`: декларативные Detour-запросы, выполняемые по
  собранному навмешу headless. Каждый запрос → результат-объект в отчёте.
  Покрывает подмножество уже реализованного в tester.
- **Зачем.** Регрессия корректности запросов и «ответ из CI»: путь A→B
  существует? какова стоимость/длина? Обобщает `testcase.zig`-формат
  (`pf`/`rc`-строки) в управляемый JSON.
- **UX / форма.**
  ```jsonc
  "queries": [
    { "type": "findPath",          "start": [x,y,z], "end": [x,y,z],
      "include": ["walk","door"], "exclude": ["disabled"],
      "expect": { "found": true } },          // опц. ожидание -> влияет на exit-code
    { "type": "findStraightPath",  "start": [...], "end": [...] },
    { "type": "raycast",           "start": [...], "end": [...] },
    { "type": "findDistanceToWall","pos":   [...] },
    { "type": "findNearestPoly",   "pos":   [...] }
  ]
  ```
  Результат на запрос: `{ found, nearest_ref, path_len, polys:[...],
  straight_points:[...], cost, dist }` (поля по типу запроса).
- **Данные.** Маппинг на существующий путь tester
  (`tool_navmesh_tester.zig`): `NavMeshQuery.init(alloc)` (стр. 120),
  `findNearestPoly` (153/157) с `ext` (полу-размеры поиска), `findPath`
  (165/179), `findStraightPath` (183/201); фильтр `QueryFilter`
  (`setIncludeFlags`/`setExcludeFlags`, 100-101; `applyCosts` из
  `area_types.zig:125`). `include`/`exclude` — имена флагов → биты через
  реестр poly-флагов (`poly_flags.bitOf`/`allMask`, `poly_flags.zig:61,77`).
- **Зависимости.**
  - **Ядро `src/*`:** `recast.detour` — `NavMeshQuery`, `QueryFilter`,
    `PolyRef`. Прямой публичный API, ядро не меняем.
  - **Scene (3.a):** реестры флагов/областей (имя→бит, area-cost). **Это
    основной потребитель выноса module-global**: имена флагов и cost берутся из
    реестра сцены, не из глобали.
  - Буферы запроса (`polys[N]`, `straight[N]`) — как в tester (фикс-ёмкости),
    лимиты в конфиг/дефолт (2048 как `tool_navmesh_tester`).

### 3.5 [P2] Машиночитаемый отчёт + exit-code

- **Что.** Единый JSON-отчёт прогона (метаданные + результаты сборки + результаты
  запросов + агрегаты) и осмысленный process exit-code. CSV — только для батча.
- **Зачем.** CI-интеграция: `recast_headless --config ci.json && echo OK`. Diff
  отчётов между ревизиями = регрессия (кластер G переиспользует).
- **UX.** `out.report` → файл; при отсутствии — stdout. Exit-code: `0` = сборка
  успешна И все `expect` совпали; `1` = провал ожидания; `2` = ошибка
  конфига/загрузки; `3` = ошибка сборки. (Точные коды — §5 Q-H6.)
- **Данные.** Сериализация `std.json.Stringify` (Zig 0.16). Метаданные:
  версия формата, путь сцены, хэш конфига (опц.), время.
- **Зависимости.** `std.json`, `io_util`. Никакого ядра/фундамента сверх 3.1-3.4.

### 3.6 [P2] CLI-обёртка / подкоманды

- **Что.** Тонкий разбор argv в `headless.zig`: `--config <f>` (основной),
  плюс быстрые override-флаги (`--sample`, `--geom`, `--out`) для одноразовых
  прогонов без конфига. Подкоманда-стиль опционально (§5 Q-H5).
- **Зачем.** Удобство ad-hoc запусков; CI обычно даёт `--config`.
- **UX.** `recast_headless --geom dungeon.obj --sample solo
  --query "findPath:0,0,0->10,0,5"` (мини-инлайн-форма — P2, опционально).
- **Данные.** Парсинг argv вручную (как `main.zig:248-266` для `--draw=`/`--cam=`),
  без зависимостей.
- **Зависимости.** Нет сверх стандартной библиотеки.

**Сводка приоритетов:** P0 = 3.1 + 3.2 (без них нет headless). P1 = 3.3 + 3.4
(основная ценность: батч и скрипт запросов). P2 = 3.5 (полировка вывода) + 3.6
(CLI-сахар).

## 4. Архитектура

> По `.agent/project_structure.md`: весь код кластера — **demo-уровень**
> (`demo/src/`), snake_case, usize-геттеры для чистых call-sites; faithful-ядро
> `src/*` не трогаем. Кластер **НЕ** передизайнивает подсистемы фундамента —
> Scene (3.a) и Persist (3.b) берутся как есть из спека-фундамента.

### 4.1 Новые файлы (`demo/src/`)

```
demo/src/
├── headless.zig            # НОВАЯ точка входа (своя main). Поднимает Scene+сэмпл
│                           #   БЕЗ glfw/GL/dvui. Оркестрация: parse argv -> load
│                           #   config -> load Scene -> build -> run queries/batch ->
│                           #   write report -> exit-code. Аналог роли main.zig, но headless.
└── headless/
    ├── config.zig          # JSON-схема прогона (§3.2): RunConfig/Settings/Batch/
    │                       #   QuerySpec + parse(std.json) + валидация/дефолты.
    ├── runner.zig          # сборка одной "ячейки": Scene.settings <- config ->
    │                       #   sample.build() -> метрики (poly/vert/ms). Один проход.
    ├── batch.zig           # декартова матрица осей (§3.3) -> цикл runner -> CSV+rows.
    ├── queries.zig         # выполнение QuerySpec[] по dt.NavMesh (§3.4):
    │                       #   маппинг type-> NavMeshQuery вызовы; имена флагов->биты;
    │                       #   агрегаты (ok/total/cost). Заимствует логику из
    │                       #   tool_navmesh_tester.zig (findPath/straight/raycast/...).
    └── report.zig          # JSON-отчёт + CSV writer (§3.5) + exit-code маппинг.
```

Размещение `headless/` как поддиректории `demo/src/` зеркалит предложенную
фундаментом структуру `demo/src/persist/`, `demo/src/render/`, `demo/src/shell/`
(фундамент §3.b/3.c/3.d) — единый паттерн «подсистема = поддиректория».

### 4.2 Точки интеграции

- **build.zig — новый exe + step.** По образцу `addDemo`
  (`build.zig:290-378`), но БЕЗ `dvui`/`zgl`/glfw-импортов и без `test_data`-as-GL:
  exe `recast_headless` с `root_source_file = demo/src/headless.zig`, импорт
  только `recast-nav` (+ `build_options`/`ztracy`-stub при желании). Отдельный
  build-step `headless` / run-step `run-headless`. **Важно:** headless НЕ должен
  тянуть `zgl.loadExtensions`/`linkSystemLibrary("opengl32")`
  (`build.zig:322-329`) — иначе теряется смысл (требование GL-драйвера). Это
  ключевая причина отдельного exe, а не флага в `recast_demo`.

- **Сэмплы без GL (блокер 3.a-уровня).** `SampleSolo`/`SampleTile`/
  `SampleTempObstacles` сейчас требуют `*DebugDrawGL` в `init`
  (`sample_solo.zig:46,61-62`), хотя build-путь (`build()`→`doBuild()`,
  `sample_solo.zig:142-360`) его НЕ использует — `dd_gl` нужен только в
  `render()`/`drawDebugMode()` (стр. 365+). Варианты (решение — §5 Q-H2):
  1. **Headless передаёт stub/`null` `DebugDrawGL`** (поле допускает отсутствие
     GL-вызовов в build) — минимальная правка демо, faithful-render не трогаем.
  2. **Расщепить сэмпл** на `build`-ядро (alloc+bctx+settings+geom→navmesh,
     без GL) и `render`-надстройку. Чище, но крупнее.
  Рекомендация спека: для MVP — вариант 1 (stub `DebugDrawGL`, который никогда не
  вызывается в headless), вынос — отдельным рефактором, если профиль потребует.

- **Scene (3.a) как единственный контейнер состояния.** `headless.zig` создаёт
  `Scene` (фундамент 3.a: `Scene.init(alloc)`), наполняет geom (Persist 3.b или
  fallback `loadMesh`), читает `areas`/`flags`/`settings` из её инстанс-полей.
  **H — главный аргумент за вынос module-global** (`area_types.zig:50`,
  `poly_flags.zig:40`): пока вынос не завершён (фундамент этап 5), headless
  ограничен «одна сцена/процесс» и опирается на переходные обёртки «активной
  сцены» (фундамент решение Q2). Спек H это явно фиксирует как зависимость, НЕ
  переопределяет дизайн реестров.

- **Persist (3.b) для загрузки сцены.** `config.scene` → вызов загрузчика 3.b
  (`scene_io`/`registry_io`/`tile_store`) для наполнения `Scene`. До готовности
  3.b — `geom`-поле конфига + `InputGeom.loadMesh` (`input_geom.zig:77`).
  **H не реализует свой формат** — переиспользует контейнер фундамента.

- **Запросы — переиспользование логики tester.** `queries.zig` не дублирует
  Detour-вызовы наивно, а выносит общую логику из `tool_navmesh_tester.zig`
  (`findNearestPoly`/`findPath`/`findStraightPath`, стр. 153-201; `getSteerTarget`/
  `smoothPath`, 547-605) в headless-вызываемые функции. Идеально — общий хелпер,
  переиспользуемый и tester'ом, и headless (избегаем дрейфа реализаций). Если
  рефактор tester рискован — headless вызывает `dt.NavMeshQuery` напрямую
  (faithful API), копируя последовательность вызовов (меньше связанность,
  риск дрейфа — в §6 R-H3).

### 4.3 Чего кластер НЕ трогает

- `main.zig` (GUI-вход) — поведение неизменно; headless — параллельный exe.
- `src/*` (faithful-ядро) — только публичный вызов `recast.detour`/`recast.recast`.
- Render (3.c) / UI-shell (3.d) — не импортируются.
- Форматы `.gset`/MSET — читаются через 3.b, не модифицируются.

## 5. Открытые вопросы / допущения (к владельцу)

> Явно НЕ угадываю — ниже формулировки и моё рабочее допущение по каждому.

- **Q-H1. Когда «множественные сцены в процессе»?** H фундаментально требует
  выноса `area_types`/`poly_flags` из module-global (фундамент 3.a этап 5).
  Стартуем ли headless в режиме «одна сцена/процесс» на переходных обёртках
  (можно сразу, до завершения выноса), а параллельный батч/несколько сцен —
  после этапа 5? *Допущение:* да — MVP headless на «одна сцена/процесс»
  (последовательный батч), параллелизм/мульти-сцена ждёт завершённого выноса.

- **Q-H2. Как сэмплу жить без GL?** Stub/`null` `DebugDrawGL` в headless (мин.
  правка) **или** расщепить сэмпл на build/render? *Допущение:* stub
  `DebugDrawGL` для MVP (никогда не вызывается в build-пути); расщепление — позже,
  если понадобится.

- **Q-H3. Строгость JSON-конфига.** Неизвестное поле = ошибка (строгий парс,
  ловит опечатки) или warning (терпимость)? *Допущение:* строгий парс +
  понятное сообщение (опечатка в CI должна падать).

- **Q-H4. Нужны ли headless-скриншоты?** Офф-скрин рендер (FBO/PBO) даёт
  «скриншот в батче», но требует GL-контекста (противоречит «headless»). Включаем
  как отдельный опциональный режим (`recast_demo --offscreen`, не в `recast_headless`)
  или вовсе откладываем? *Допущение:* **откладываем** — батч даёт метрики/таблицы,
  не картинки; скриншоты — отдельный поздний вопрос (возможно через GUI-exe с
  off-screen, не через headless-exe).

- **Q-H5. REPL/подкоманды/инлайн-запрос?** Задание упоминало «мини-скрипт/REPL».
  Свожу к JSON-сценарию (YAGNI). Нужен ли всё же построчный stdin-REPL
  (`> findPath 0,0,0 10,0,5`) и/или подкоманды (`headless build`/`headless query`)
  в стартовом scope, или JSON-конфига достаточно? *Допущение:* JSON-конфиг +
  тонкие override-флаги (§3.6) достаточно; REPL — отдельный поздний P2, если
  будет спрос.

- **Q-H6. Семантика exit-code.** Предлагаю `0`=всё ок, `1`=провал `expect`-
  ожидания, `2`=ошибка конфига/загрузки, `3`=ошибка сборки. Подходит для вашего
  CI, или нужна иная гранулярность (напр. отдельный код на «частичный успех
  батча»)? *Допущение:* схема 0/1/2/3 выше.

- **Q-H7. Tile/Temp в headless.** Solo тривиален. Tile-сборка
  (`sample_tile.zig`) и Temp Obstacles (`sample_temp_obstacles.zig`, tilecache)
  — нужны ли в MVP, или MVP = только Solo, а tile/temp — следующий этап?
  *Допущение:* MVP = Solo + Tile (оба чисто строят навмеш); Temp Obstacles
  (динамика/tilecache) — позже (пересекается с кластером J).

- **Q-H8. Где общая логика запросов?** Выносим Detour-вызовы из
  `tool_navmesh_tester` в общий хелпер (риск задеть GUI-tester) или дублируем в
  `queries.zig` (риск дрейфа)? *Допущение:* для MVP — отдельная реализация в
  `queries.zig` поверх публичного `dt.NavMeshQuery` (меньше связанность с GUI),
  с TODO на объединение, если дрейф станет проблемой.

## 6. Риски

- **R-H1 — Module-global реестры (наследие фундамента R3).** `area_types`/
  `poly_flags` глобальны (`area_types.zig:50`, `poly_flags.zig:40`); до выноса
  (3.a этап 5) headless физически ограничен одной сценой/процессом, а
  параллельный батч **некорректен** (ячейки делят глобаль). *Митигация:*
  MVP — последовательный батч на одной сцене; параллелизм гейтить за завершённым
  выносом; явно документировать ограничение в `--help`.

- **R-H2 — Сэмпл тянет `DebugDrawGL` в конструкторе.** Даже без рендера сэмпл
  требует `*DebugDrawGL` (`sample_solo.zig:61`). Stub-обход хрупок, если
  где-то в build-пути появится GL-вызов. *Митигация:* stub, который `@panic`-ит
  при вызове любого метода (ловит регресс «GL в build-пути» на тесте); long-term
  — расщепление build/render (§5 Q-H2).

- **R-H3 — Дрейф логики запросов.** Если `queries.zig` дублирует
  последовательность вызовов tester, они разойдутся при правке одного из них
  (баг «headless и GUI дают разный путь»). *Митигация:* регрессионный тест
  (`test/integration/`), сверяющий headless-результат с эталоном/GUI на фикс-сцене;
  по возможности — общий хелпер (§5 Q-H8).

- **R-H4 — Persist (3.b) не готов на старте H.** H зависит от загрузки
  `.recastscene/`, которой ещё нет (фундамент этап 4). *Митигация:* fallback на
  `InputGeom.loadMesh(.obj)` + конфиг-settings; полная загрузка — после 3.b.
  Спек явно помечает это как зависимость от ещё-не-реализованного фундамента.

- **R-H5 — Зависимость build от cwd/ассетов.** GUI ищет `test_data` рядом с exe
  (`build.zig:362-366`, FEATURES §11). Headless должен брать пути из конфига
  (абсолютные/относительные явно), не полагаясь на сканирование каталога ассетов
  при старте. *Митигация:* все пути — из конфига; no implicit asset scan.

- **R-H6 — Раздувание области demo без shell (3.d).** H добавляет отдельный
  вход — это снимает риск раздувания `main.zig` (мы его не трогаем), но создаёт
  второй путь сборки сцены (headless vs GUI). *Митигация:* общий код через
  `Scene`/`runner`, а не копипаст конвейера сборки.

- **R-H7 — Zig 0.16 `std.json` API.** Парс/сериализация на движущемся 0.16
  std (как и риск R1 фундамента про `Io`). *Митигация:* тонкая обёртка над
  `std.json` в `config.zig`/`report.zig`, локализующая возможные API-сдвиги.

## 7. Этапы реализации (порядок)

> Порядок выбран так, чтобы каждый шаг давал работающий артефакт и не зависел от
> ещё-не-готовых частей фундамента раньше необходимого. **Предусловие всего
> кластера:** фундамент этапы 1-2 (U-shell не нужен; нужен **Scene-каркас 3.a**,
> фундамент этап 2) — H читает `Scene`. Persist (этап 4 фундамента) и полный
> вынос module-global (этап 5 фундамента) — нужны позже (см. ниже).

1. **Headless entry + Solo-сборка из конфига (P0).** `headless.zig` + `config.zig`
   + `runner.zig`: parse JSON → `Scene` (fallback `loadMesh(.obj)`, без Persist) →
   `SampleSolo.build()` (через stub `DebugDrawGL`, §4.2) → метрики (poly/vert/ms)
   в stdout-JSON. Новый exe/step в `build.zig` (БЕЗ glfw/GL). *Даёт:* «собери
   навмеш из терминала без окна» + базовый отчёт сборки. *Зависит от:* Scene-каркас
   (3.a этап 2).

2. **Сценарий запросов (P1).** `queries.zig` + расширение `report.zig`: массив
   `findPath`/`findStraightPath`/`raycast`/`findDistanceToWall`/`findNearestPoly`
   по собранному навмешу; имена флагов→биты из реестра; агрегаты ok/total/cost;
   `expect`→exit-code. Регрессионный тест против GUI/эталона (R-H3). *Даёт:*
   «ответь из CI, есть ли путь A→B». *Зависит от:* реестры флагов из Scene (3.a).

3. **Батч-матрица (P1) + Tile-сэмпл.** `batch.zig`: декартово произведение осей →
   цикл `runner` → CSV+JSON-таблица. Добавить `SampleTile` в `runner` (Q-H7).
   Последовательный (один процесс, общий реестр — корректно). *Даёт:* «матрица
   параметров → таблица метрик». *Зависит от:* этапы 1-2.

4. **Полная загрузка сцены через Persist (P0/P1) + полировка вывода (P2).**
   Заменить fallback `loadMesh` на загрузку `.recastscene/` (3.b: areas/flags/
   volumes/offmesh + tiles). Финализировать JSON-отчёт/CSV/exit-codes (3.5),
   CLI-override-флаги (3.6). *Зависит от:* **Persist готов (фундамент этап 4).**

5. **(Опц., по потребности) Параллельный батч + Temp Obstacles.** Threadpool по
   ячейкам — **только после полного выноса module-global** (фундамент этап 5) и
   thread-safe реестров; Temp Obstacles/tilecache (пересечение с кластером J).
   YAGNI до подтверждённой потребности в скорости батча.

**Критический путь H:** фундамент(этап2 Scene) → H-этап1 → H-этап2 → H-этап3.
H-этап4 ждёт Persist (фундамент этап4); H-этап5 ждёт вынос module-global
(фундамент этап5) — оба отмечены как зависимости от ещё-не-готового фундамента.
