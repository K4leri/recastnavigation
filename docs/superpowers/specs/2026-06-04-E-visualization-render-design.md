## 1. Цель + ценность для developer-user

Сейчас визуализация — это faithful-порт `RecastDemo`: фиксированный `DrawMode`
enum на сэмпл (`sample_solo.zig:22`, `sample_tile.zig:22`), один способ окраски
navmesh (по area, через хук `dd_gl.area_to_col = sample.sampleAreaToCol`,
`sample_solo.zig:365`), отрисовка целиком (`debugDrawNavMesh` обходит **все**
тайлы, `detour_debug.zig:31-37`), без подписей полигонов, без легенды, без
изоляции слоёв/высот, без перелёта к объекту.

Для инженера-пользователя это «смотрю на цветное пятно». E превращает рендер в
**аналитический инструмент**:

- **Многоэтажность читаема.** Сечение по высоте (clipping plane) и изоляция
  слоёв tile-cache разводят перекрывающиеся этажи, которые сейчас сливаются в
  Z-fighting (`debugDrawNavMeshPoly` уже бьётся `depthMask(false)` —
  `detour_debug.zig:132`).
- **Семантика видна.** Цветовые схемы по area/flags/height/component/cost/tile
  отвечают «какие зоны дорогие», «что несвязно», «где границы тайлов» — вместо
  единственной area-раскраски.
- **Идентификация мгновенна.** Подписи (poly-ref/координаты/cost) и легенда
  убирают `std.debug.print` инспекцию полигона (фундамент §1 ссылается на
  `main.zig:480-491`).
- **Навигация по большому мешу.** Минимап + перелёт-к-тайлу/полигону-по-id
  заменяют ручную орбиту камеры на больших сценах.
- **Воспроизводимый показ.** Запись последовательности кадров даёт
  detached-репро для багрепорта/обзора (смежно с кластером I).

Ценность: те же данные (`dt.NavMesh`, промежуточные структуры Recast), но
**избирательно, послойно, подписанно и с перелётом** — debug перестаёт быть
«глазами по пикселям».

## 2. Scope (in / out / YAGNI)

### Входит (in)

- **Сечение / clipping plane** по высоте (Y) — резать видимость navmesh/геометрии
  выше/ниже плоскости; интерактивный слайдер высоты; режим «slab» (полоса
  ±толщина). Реализация — **предикат видимости при per-poly обходе** (как 3.c
  решил для изоляции), плюс опциональный GL-clip как ускорение для input-mesh.
- **Изоляция / прозрачность слоёв** — show-only / dim-others по
  {tile | area | flags | region(если доступен) | layer(tile-cache)}; wireframe ↔
  solid toggle; скрытие input-mesh/navmesh/overlay-групп. Наполнение 3.c
  `isolation.zig`.
- **Цветовые схемы** — `ColorScheme { area, flags, height, tile, component, cost }`
  поверх каркаса 3.c `color_scheme.zig`. **area/flags/tile/height — из коробки**
  (данные есть в baked-навмеше); **region/component/cost — производные**
  (см. §5 Q1 — критично).
- **Подписи / оверлеи** — poly-ref (hex), координаты центроида, area/flags/cost
  над полигоном; toggle плотности (только под курсором / все видимые / none).
  Наполнение 3.c `overlay.zig` поверх `ui.screenTextEx` (`ui.zig:51`) +
  `cam.worldToScreen` (`camera.zig:80`).
- **Легенда** — 2D-свотчи цвет→подпись активной схемы; наполнение 3.c
  `legend.zig`.
- **Минимап / обзор** — top-down 2D-проекция тайловой сетки + bbox сцены +
  маркер камеры; клик/ввод id → **перелёт камеры** к тайлу (tx,ty) или полигону
  (poly-ref) через `cam.reset`/прямую установку `cam.pos`/`eulers`.
- **Запись клипа / последовательности кадров** — захват N кадров (PPM/PNG) по
  траектории (орбита, как `--bench`, `main.zig:308-335`) или по live-вводу;
  выгрузка в каталог. Headless-friendly (без UI-зависимости в самом захвате).

### Не входит (out)

- Замена `debug_draw_gl.zig` бэкенда, шейдеров, `dvui`/`zgl`. E рисует **через
  существующий** `DebugDraw` vtable.
- Передизайн подсистем фундамента 3.a–3.d. E их **потребляет**.
- Новые debug-режимы *стадий Recast* (rasterize/regions/contours визуализация) —
  это **кластер B** (intro­spection build-конвейера, фундамент §4: B=R●●). E
  работает с *готовым navmesh + базовой геометрией*, не со стадиями.
- Видео-кодек (mp4/h264). Захват — **последовательность кадров** (изображения),
  кодирование оставляем внешнему `ffmpeg` (документируем команду).

### YAGNI (пока не делаем)

- Произвольная ориентация clipping-плоскости (не только Y). Многоэтажность —
  это Y-сечение; косые плоскости — без подтверждённой потребности.
- Несколько одновременных clipping-плоскостей / CSG.
- Объёмный/теневой рендер, PBR, освещение. Туман уже есть
  (`debug_draw_gl.zig:87`), его достаточно.
- Анимация интерполяции цветовых схем, тепловые градиенты с настраиваемыми
  стопами — фиксированные палитры из `src/debug/debug_draw.zig`
  (`intToCol`/`lerpCol`) достаточно.
- Встроенный видеоплеер/таймлайн записанных клипов.
- 3D-гизмо перетаскивания clipping-плоскости — слайдер высоты проще и точнее.

## 3. Фичи (приоритизированы)

Приоритет: **P0** разблокирует чтение многоэтажности и семантики (главная боль);
**P1** — идентификация/навигация; **P2** — экспорт/запись.

---

### P0-1. Цветовые схемы (`ColorScheme`)

- **Что.** Enum `ColorScheme { area, flags, height, tile, component, cost }` и
  функция выбора цвета полигона. area/flags/tile/height — **сразу**;
  region/component/cost — производные (см. зависимости и §5 Q1).
- **Зачем.** Единственная area-раскраска не отвечает на «дорого/несвязно/где
  границы тайла/какой этаж». Заменяет хардкод `sampleAreaToCol`
  (`sample.zig:53`) и `vtAreaToCol` (`debug_draw_gl.zig:391`) единым выбором.
- **UX.** Дропдаун «Color By» в панели Properties (рядом с Debug Settings).
  Смена схемы — мгновенная (перекраска при следующем обходе тайлов). Привязанная
  легенда (P1-3) обновляется автоматически.
- **Данные.**
  - `area` — `Poly.getArea()` (`navmesh.zig:50`) → цвет из `AreaRegistry`
    (фундамент 3.a, сейчас `area_types.colorFor`). **Уже работает.**
  - `flags` — `Poly.flags` (`navmesh.zig:22`) → цвет по доминантному биту из
    `FlagRegistry`. Данные в baked-навмеше есть.
  - `tile` — `decodePolyId(ref).tile` → `intToCol(tile_num)` (паттерн уже в
    `drawMeshTile`, `detour_debug.zig:160-161`; `COLOR_TILES`-флаг). **Есть.**
  - `height` — Y центроида полигона (из `tile.verts`/`detail_verts`),
    нормировка по bbox сцены → градиент `lerpCol`. Чисто геометрия, **есть**.
  - `cost` — **НЕ в baked-навмеше** (`Poly` хранит только area, не cost,
    `navmesh.zig:11-57`). Производное: `cost = AreaRegistry.cost(getArea())`.
    Доступно через реестр (3.a). Внятно при ≤64 area-типов.
  - `component` — **НЕ в baked-навмеше**. Требует connectivity flood-fill по
    линкам полигонов (тот же обход, что `tool_prune.zig`, FEATURES §4.2) →
    `component_id[ref]`. Считается demo-уровнем, кешируется по навмешу.
  - `region` — **отсутствует в baked Detour Poly** (region — стадия Recast, не
    переживает poly-mesh bake). Либо берём из промежуточного `PolyMesh.regs`
    (только Solo, держит intermediates), либо считаем эрзац через component.
    **Открытый вопрос §5 Q1.**
- **Зависимости.** 3.c `color_scheme.zig` (каркас). Ядро `src/detour/navmesh.zig`
  (`Poly`, `decodePolyId`). 3.a `AreaRegistry`/`FlagRegistry` (cost/flag-цвета).
  Для region/component — обход линков (`tile.links`, `Poly.first_link`); для
  region из intermediates — `src/recast` `PolyMesh.regs` (только Solo).

### P0-2. Сечение по высоте (clipping plane) + слой-изоляция

- **Что.** Горизонтальная Y-плоскость: показать полигоны/геометрию выше/ниже/в
  полосе [y−t, y+t]. Плюс изоляция по {tile|area|flags|layer}: «show only» /
  «dim others» (затемнение через `transCol`/`darkenCol`, `debug_draw.zig`).
- **Зачем.** Многоэтажные меши (паркинги, здания) сливаются в Z-fight — глаз не
  разделяет этажи. Сечение и dim-others разводят их.
- **UX.** Секция «Clipping / Isolation» в Properties: слайдер высоты Y (диапазон
  из bbox сцены `geom.bmin[1]..bmax[1]`), радио {off | above | below | slab} +
  слайдер толщины слэба; режим изоляции {none | show-only | dim-others} +
  выбор ключа и значения (напр. tile (tx,ty), area-id, layer-id).
- **Данные.** Y центроида полигона vs плоскость → булев предикат видимости,
  применяется **в обходе** (3.c решил: «не GL-clip, а per-poly предикат»,
  фундамент 3.c). Для input-mesh (десятки тыс. треугольников) — опциональный
  GL clip-plane как ускорение (вторичный путь).
- **Зависимости.** 3.c `isolation.zig` (каркас предиката). Требуется **точка
  расширения в обходе** `drawMeshTile`/`debugDrawNavMesh` — см. §4 (центральное
  архитектурное решение: per-poly callback/предикат). 3.a `Scene` для bbox/слоёв.
  Layer-изоляция опирается на tile-cache (`HeightfieldLayerSet.layerCount()`,
  CLAUDE.md) — релевантно сэмплу Temp Obstacles.

### P1-1. Wireframe ↔ solid + видимость групп

- **Что.** Toggle отрисовки navmesh как заливка (текущее, `.tris` +
  `transCol 64`) vs только рёбра (`.lines`, как `drawPolyBoundaries`,
  `detour_debug.zig:219`); чекбоксы видимости групп: input-mesh / navmesh /
  off-mesh / convex-volumes / overlay-подписи.
- **Зачем.** Solid скрывает геометрию под собой; wireframe даёт увидеть слои
  насквозь. Скрытие групп убирает визуальный шум (сейчас off-mesh/convex
  рисуются **всегда**, FEATURES §7).
- **UX.** Чекбоксы в «Clipping / Isolation» или отдельная секция «Layers».
- **Данные.** Флаги видимости (demo-state). Wireframe = вызвать
  `drawPolyBoundaries`-стиль вместо/поверх заливки.
- **Зависимости.** 3.c (общий стиль рисования рёбер). UI-shell (3.d) панель.
  Группы off-mesh/convex рисуются сэмплом — нужен флаг-гейт (минимальная правка
  сэмплов, не ядра).

### P1-2. Подписи / оверлеи (poly-ref, координаты, cost)

- **Что.** Worldspace-текст над полигоном: poly-ref (hex), центроид (x,y,z),
  area-имя, cost; режим плотности {none | hovered | all-visible}.
- **Зачем.** Убирает stderr-инспекцию (`main.zig:480-491`); «что это за полигон»
  читается в сцене.
- **UX.** Дропдаун «Labels»: {none | hovered | all}. all — только для видимых
  (после clipping/isolation) полигонов, иначе тысячи строк. hovered — пикинг
  лучом (как инспекция Disabled-инструмента, FEATURES §4) → подпись под курсором.
- **Данные.** Центроид из вершин полигона; `cam.worldToScreen`
  (`camera.zig:80`) → `ui.screenTextEx` (`ui.zig:51`). Отсев по `sp.z∈[0,1]` (уже
  паттерн `main.zig:951`). Cull по числу: при all и >N видимых — авто-понижение
  до hovered (защита перфа).
- **Зависимости.** 3.c `overlay.zig` (обобщает разрозненные `screenTextEx` из
  `main.zig:894-985`). Пикинг — `cam.pickRay` (`camera.zig:68`) + `findNearestPoly`
  (Detour).

### P1-3. Легенда

- **Что.** 2D-панель свотчей (цвет→подпись) для активной `ColorScheme`.
- **Зачем.** Без легенды цветовая схема — ребус (особенно height/cost-градиент,
  component-id).
- **UX.** Угловой оверлей (как hint-текст `main.zig:968-984`), скрываемый
  чекбоксом. Для дискретных схем (area/flags/tile/component) — список свотчей;
  для непрерывных (height/cost) — градиент-бар с min/max подписями.
- **Данные.** Запрос у активной схемы списка (label,color) или (min,max,
  градиент). Реестры area/flags дают имена (3.a).
- **Зависимости.** 3.c `legend.zig`. `ColorScheme` (P0-1) как источник entries.
  dvui-бокс + `ui.screenText`.

### P1-4. Минимап / обзор + перелёт по id

- **Что.** Top-down 2D-виджет: bbox сцены, тайловая сетка (для Tile/Temp),
  маркер позиции/направления камеры, точки off-mesh/convex. Поле ввода
  tile (tx,ty) или poly-ref (hex) → **перелёт камеры** к объекту.
- **Зачем.** На больших тайловых мешах ручная орбита к нужному тайлу медленна;
  «покажи тайл 12,7» / «покажи polyref 0x...» — частый debug-запрос.
- **UX.** Скрываемое окно «Minimap» (через UI-shell panel-дескриптор 3.d).
  Клик по минимапу — перелёт к точке. Поле id + кнопка «Go».
- **Данные.** Тайловая сетка из `NavMesh` (origin/tileWidth/tileHeight,
  параметры в header). Камера: `cam.reset(bmin,bmax)` для bbox или прямая
  установка `cam.pos`/`cam.eulers` для центрирования на центроиде тайла/полигона
  (паттерн `main.zig:279-281,1078`). poly-ref → `decodePolyId` → центроид.
- **Зависимости.** `camera.zig` (reset/pos/eulers). `src/detour/navmesh.zig`
  (decode, tile-параметры). UI-shell (3.d) для окна. 3.a `Scene` (bbox).

## 4. Архитектура

### Размещение файлов (`.agent/project_structure.md`)

E — **demo-уровень**, поверх фундамента 3.c. Новые файлы — в каталоге render
фундамента (3.c определил `demo/src/render/`), E **наполняет** заявленные там
модули и добавляет E-специфичные:

```
demo/src/render/            # каталог из фундамента 3.c — E его наполняет
├── color_scheme.zig        # 3.c КАРКАС → E реализует enum+colorForPoly для
│                           #   area/flags/height/tile; + component (flood-fill кеш)
│                           #   + cost (из AreaRegistry); region — по §5 Q1.
├── isolation.zig           # 3.c КАРКАС → E реализует Y-clipping предикат + slab +
│                           #   show-only/dim-others по ключу.
├── overlay.zig             # 3.c КАРКАС → E реализует poly-ref/coord/cost подписи,
│                           #   режим плотности, hovered-пикинг.
├── legend.zig              # 3.c КАРКАС → E реализует свотчи/градиент-бар по схеме.
├── poly_visit.zig          # НОВЫЙ (E): обход полигонов навмеша с предикатом
│                           #   видимости + per-poly color-callback. Не дублирует
│                           #   drawMeshTile, а параметризует его (см. ниже).
├── minimap.zig             # НОВЫЙ (E): top-down проекция + клик→перелёт.
└── capture.zig             # НОВЫЙ (E): glReadPixels → write_atomic кадры.
```

### Центральное архитектурное решение: точка расширения обхода

Сегодня `debugDrawNavMesh` (ядро `src/debug/detour_debug.zig:31`) **жёстко**
обходит все тайлы и красит каждый полигон через `dd.areaToCol` (`:180`) — без
предиката видимости и без выбора схемы. Цвет переопределяется только глобальным
`area_to_col` (`debug_draw_gl.zig:112`), которого мало для height/cost/component
(нужен контекст полигона, не только area — это зафиксировано в фундаменте 3.c).

**Faithful-ядро `src/debug/detour_debug.zig` НЕ меняем** (CLAUDE.md: 1-в-1 с
upstream). Вместо этого E даёт **demo-уровневый обходчик** `poly_visit.zig`,
который дублирует *структуру* обхода `drawMeshTile` (тот же проход по
detail-tris, `detour_debug.zig:189-202`), но принимает:

- `visible_fn(ctx, ref, poly, tile) bool` — предикат изоляции/clipping;
- `color_fn(ctx, ref, poly, tile) u32` — активная `ColorScheme`.

Тогда:

- **Стандартный режим** (faithful) — продолжает звать `debugDrawNavMesh` ядра как
  сейчас (сэмплы не трогаем для дефолтного рендера).
- **E-режим** (схема/clipping/isolation активны) — сэмпл зовёт
  `poly_visit.drawNavMeshFiltered(dd, mesh, ctx)` вместо ядрового. Это
  **единственная** новая точка интеграции в render-пути; решение «параметризовать
  копией, не править ядро» прямо следует из CLAUDE.md + фундамент 3.c.

> Альтернатива (расширить `area_to_col` до `poly_to_col(ref,*Poly)`) **отвергнута**:

## 5. Открытые вопросы / допущения к владельцу

**Q1 (КРИТИЧНО) — источник region/component для схем.** Baked Detour `Poly`
(`navmesh.zig:11-57`) хранит **только** area+type+flags — **region и component
НЕ переживают bake**. Варианты:
(a) `region` брать из intermediate `PolyMesh.regs` — но это есть **только у Solo**
(держит промежуточные структуры; Tile/Temp их сбрасывают); (b) `component`
считать flood-fill по линкам на demo-уровне (работает для всех сэмплов, кеш по
навмешу); (c) отказаться от `region`, оставить `component` как замену.
*Допущение:* делаем **component через flood-fill** (универсально), `region` —
**только для Solo из intermediates**, иначе схема region недоступна (серый/скрыта).
Подтвердить: достаточно ли component, или region обязателен и для Tile?

**Q2 — clipping: предикат vs GL-плоскость.** Фундамент 3.c постановил «предикат
видимости, не GL-clip». *Допущение:* navmesh — предикат (per-poly, дёшево);
input-mesh (десятки тыс. tris) — опциональный GL `gl_ClipDistance`/discard как
ускорение. Подтвердить: нужен ли clipping и для input-mesh, или хватит navmesh?

**Q3 — плотность подписей по умолчанию.** all-visible на большом меше = тысячи
`screenTextEx` (перф). *Допущение:* дефолт `hovered`; `all` авто-понижается до
`hovered` при >N видимых (N≈300). Подтвердить порог/поведение.

**Q4 — формат записи кадров.** PNG требует энкодер (внешняя Zig-зависимость или
порт); PPM — тривиально, но крупно и не везде открывается. *Допущение:* стартуем
с **PPM** (нулевая зависимость), PNG — опционально позже; видео — внешний
`ffmpeg` (документируем). Подтвердить: нужен ли PNG сразу?

**Q5 — capture зависит от ещё-не-готового 3.b.** Durable-запись кадров хочет
`write_atomic.zig` (фундамент этап 4). *Допущение:* P2-1 идёт **после** 3.b;
до этого — деградация на `io_util.writeWholeFile` (не durable, но рабоче).
Подтвердить порядок.

**Q6 — граница E vs кластер B.** Визуализация *стадий* Recast
(voxels/regions/contours) — у B (фундамент §4). E берёт готовый navmesh + базовую
геометрию. *Допущение:* схема `region` E использует уже-имеющиеся `PolyMesh.regs`
(не строит стадию заново — это B). Подтвердить границу.

**Q7 — перелёт камеры: анимация или прыжок.** *Допущение:* мгновенный прыжок
(`cam.pos`/`eulers` напрямую) — проще, детерминированно (важно для headless/
capture). Подтвердить: нужна ли плавная интерполяция?

## 6. Риски

- **R-E1 — region недоступен вне Solo (Q1).** Центральная схема может оказаться
  неполной. *Митигация:* component-flood-fill как универсальная замена; region
  помечаем «Solo-only» в UI.
- **R-E2 — перф per-poly обхода с предикатом+схемой на больших навмешах**
  (фундамент R6). Дублирующий обход `poly_visit` добавляет per-poly работу.
  *Митигация:* предикаты дешёвые (сравнение Y/area/tile-id); переиспускаем
  батчинг VBO `debug_draw_gl`; контролируем `draw_calls`/`verts_uploaded`
  (`:103-104`); кеш component_id и centroid по навмешу (инвалидация по rebuild).
- **R-E3 — подписи (worldspace-текст) дорогие.** Тысячи `renderText`/кадр
  (`ui.zig:59`) роняют fps. *Митигация:* Q3 (hovered-дефолт, авто-cull,
  отсев `sp.z∉[0,1]`).
- **R-E4 — capture зависит от 3.b (Q5).** Если E стартует раньше фундамент-этапа
  4 — нет durable-записи. *Митигация:* деградация на `writeWholeFile`; durable —
  когда 3.b готов.
- **R-E5 — расхождение `poly_visit` с ядровым `drawMeshTile`.** Копия структуры
  обхода может разойтись с upstream-поведением (detail-tris, boundaries).
  *Митигация:* `poly_visit` строит **тот же** detail-tri проход
  (`detour_debug.zig:189-202`); регресс-тест «дефолтная схема + предикат
  always-true == ядровый `debugDrawNavMesh`» (визуальный/счётчики вершин).
- **R-E6 — `area_to_col`-хук конфликтует со схемами.** Сэмплы ставят
  `dd_gl.area_to_col` (`sample_solo.zig:365`). В E-режиме цвет идёт из
  `poly_visit`, хук не используется → возможна путаница. *Митигация:* в E-режиме
  не полагаемся на `area_to_col`; чётко разделить пути (ядровый vs E).
- **R-E7 — минимап/перелёт на Solo (1 тайл) бессмысленны.** *Митигация:*
  минимап показывает bbox+camera всегда, тайловую сетку — только Tile/Temp;
  перелёт по poly-ref работает везде.
- **R-E8 — раздувание `main.zig` (фундамент R5).** Без UI-shell каждая секция
  растит монолит. *Митигация:* секции/окна E — через 3.d panel-дескрипторы.

## 7. Этапы реализации (порядок)

Порядок: сначала то, что снимает главную боль (чтение семантики/многоэтажности)
и опирается на **уже готовые** части фундамента (3.c-каркас, 3.a-данные), затем
навигация, затем экспорт (ждёт 3.b).

1. **E0 — `color_scheme.zig` (area/flags/tile/height).** Только схемы на данных,
   что **уже есть** в baked-навмеше. Дропдаун «Color By». Без обхода-рефактора:
   area/flags/tile — через расширение пути цвета; height — требует centroid →
   тянет шаг E1. *Разблокирует:* чтение семантики сразу.
2. **E1 — `poly_visit.zig` (точка расширения обхода).** Demo-обходчик с
   `visible_fn`+`color_fn`, регресс-тест против ядрового `debugDrawNavMesh`
   (R-E5). Подключение в сэмплы веткой. *Разблокирует:* height/cost/component-
   схемы и clipping/isolation (всем нужен per-poly контекст+предикат).
3. **E2 — `isolation.zig` (Y-clipping + slab + show-only/dim-others).** Поверх
   E1-предиката. Секция Clipping/Isolation. *Разблокирует:* многоэтажность.
4. **E3 — схемы component (flood-fill кеш) + cost (AreaRegistry); region (Solo).**
   Решение Q1. *Разблокирует:* связность/стоимость.
5. **E4 — wireframe/solid + видимость групп (P1-1).** Дёшево поверх E1/E2.
6. **E5 — `overlay.zig` (подписи) + `legend.zig`.** Поверх схем (нужна активная
   схема для легенды) и `worldToScreen`. Q3 (плотность).
7. **E6 — `minimap.zig` + перелёт по id.** Независим от схем; через UI-shell.
8. **E7 — `capture.zig` (P2-1).** **После** фундамент-этапа 4 (3.b
   `write_atomic`), иначе деградация (Q5). CLI `--capture`.

Критический путь главной ценности: **E0 → E1 → E2** (семантика + многоэтажность).
E3–E5 наращивают глубину; E6 — навигация; E7 — экспорт (ждёт 3.b).
